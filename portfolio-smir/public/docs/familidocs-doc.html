1. Contexte et objectifs

FamiliDocs est un coffre-fort numerique familial concu pour centraliser, organiser et securiser l'ensemble des documents administratifs d'un foyer. L'application repond a un besoin concret et universel : offrir un espace unique, protege et accessible ou chaque membre de la famille peut stocker, retrouver et partager ses documents importants.

Le projet s'inscrit dans le cadre du BTS SIO option SLAM, epreuve E5 (Conception et maintenance de solutions informatiques). Il constitue le projet principal du candidat et illustre la maitrise complete du cycle de developpement d'une application web, de la conception a la mise en production.

Objectifs fonctionnels

  • Centralisation : regrouper tous les documents administratifs (impots, sante, banque, logement, identite) dans un espace unique et structure
  • Organisation : classer par categories, sous-dossiers, tags et niveaux de confidentialite pour retrouver instantanement n'importe quel document
  • Securisation : proteger les donnees sensibles par chiffrement AES, hachage bcrypt, gestion stricte des sessions et controle d'acces granulaire
  • Partage controle : permettre le partage de documents entre membres de confiance (conjoint, enfants, aidants) avec des permissions fines et temporaires
  • Suivi des echeances : alerter les utilisateurs sur les documents expirants, les taches en cours et les deadlines administratives

Public cible

  • Parents ages : personnes peu a l'aise avec le numerique, necessitant une interface simple et intuitive
  • Familles : foyers souhaitant centraliser les documents de tous les membres
  • Aidants familiaux : personnes accompagnant un proche dans ses demarches administratives
  • Particuliers : toute personne souhaitant securiser et organiser ses papiers importants

L'interface a ete pensee pour etre accessible, claire et non intimidante, avec une navigation intuitive et des libelles explicites.

2. Architecture technique

Pattern MVC adapte a Flask

L'application suit le pattern Model-View-Controller adapte aux conventions Flask :

  • Models : classes SQLAlchemy declaratives representant les 9 tables de la base de donnees, avec relations, validations et methodes metier
  • Views : templates Jinja2 organises par module fonctionnel, heritant d'un layout de base avec navigation dynamique
  • Controllers : Blueprints Flask regroupant les routes par domaine (auth, documents, admin, tasks, tags, notifications)
  • Services : couche metier separee pour le chiffrement, la gestion des fichiers et les notifications

4. Fonctionnalites detaillees

4.a Gestion des utilisateurs

Le systeme gere trois niveaux de roles avec des permissions hierarchiques :

RoleDroitsUsage type
adminAcces total : gestion des utilisateurs, logs systeme, sauvegardes, configuration globaleAdministrateur du foyer
userCRUD sur ses propres documents, dossiers et taches. Partage vers d'autres utilisateursMembre standard de la famille
trustedDroits identiques a user + acces aux documents partages avec permissions etenduesPersonne de confiance, aidant

Securite de connexion :

  • Rate limiting : maximum 5 tentatives echouees par adresse IP
  • Blocage automatique pendant 15 minutes apres depassement
  • Journalisation de chaque tentative (reussie ou echouee) avec IP et User-Agent
  • Politique de mot de passe stricte : 8 caracteres minimum, au moins 1 majuscule, 1 minuscule, 1 chiffre et 1 caractere special

4.b Gestion documentaire

Coeur fonctionnel de l'application, la gestion des documents couvre l'ensemble du cycle de vie des fichiers administratifs.

Formats supportes :

  • Documents : PDF, DOC, DOCX (Word), XLS, XLSX (Excel), TXT
  • Images : JPG, JPEG, PNG, GIF, BMP, WEBP
  • Limite de taille : 16 Mo par fichier

Dossiers hierarchiques : classement par categories predefinies a la creation du compte :

  • Administratif : pieces d'identite, impots, actes officiels
  • Sante : ordonnances, resultats d'analyses, carnet de sante
  • Banque : releves, contrats, justificatifs
  • Logement : bail, quittances, diagnostics, assurances
  • Autres : tous les documents ne relevant pas des categories precedentes

L'utilisateur peut creer des sous-dossiers a l'interieur de chaque categorie pour affiner son classement.

Niveaux de confidentialite :

  • public : visible par tous les membres du foyer
  • private : visible uniquement par le proprietaire
  • restricted : visible uniquement par les utilisateurs ayant recu une permission explicite

4.c Versioning des documents

Chaque document dispose d'un historique complet de ses versions, permettant la tracabilite et la recuperation de versions anterieures.

  • Upload de nouvelle version : ajout d'un fichier actualise avec commentaire obligatoire decrivant les modifications
  • Historique chronologique : consultation de toutes les versions avec date, auteur et commentaire
  • Restauration : possibilite de restaurer n'importe quelle version anterieure comme version courante
  • Telechargement individuel : chaque version peut etre telechargee independamment
  • Numerotation automatique : numero de version incremental gere par le systeme

Ce mecanisme est particulierement utile pour les documents administratifs mis a jour periodiquement (avis d'imposition annuel, attestation d'assurance, etc.).

4.d Tags et recherche avancee

Systeme de tags :

  • Creation de tags personnalises avec couleur hexadecimale au choix
  • Association multiple : un document peut porter plusieurs tags, un tag peut etre applique a plusieurs documents
  • Filtrage rapide par tag depuis l'interface de listing
  • Gestion complete (creation, modification, suppression) avec verification d'integrite

Recherche multi-criteres :

  • Texte libre : recherche dans le nom du document et le nom original du fichier
  • Type de fichier : filtrage par extension (PDF, images, Word, Excel...)
  • Dossier : filtrage par categorie ou sous-dossier
  • Tags : filtrage par un ou plusieurs tags
  • Dates : plage de dates de creation ou de modification
  • Confidentialite : filtrage par niveau (public, prive, restreint)

Recherche globale AJAX : barre de recherche dans la navbar avec suggestions en temps reel, sans rechargement de page.

5. Securite

La securite constitue un pilier central de FamiliDocs, l'application gerant des documents administratifs sensibles. Chaque couche de l'architecture integre des mecanismes de protection specifiques.

Authentification et mots de passe

  • Hachage bcrypt : chaque mot de passe est hache avec un salt aleatoire unique et un cout de 12 rounds, rendant les attaques par force brute et par tables arc-en-ciel impraticables
  • Politique de mot de passe : minimum 8 caracteres, au moins 1 majuscule, 1 minuscule, 1 chiffre et 1 caractere special
  • Rate limiting : maximum 5 tentatives de connexion echouees par adresse IP, suivi d'un blocage automatique de 15 minutes
  • Sessions securisees : duree de vie de 2 heures, cookie HttpOnly (inaccessible au JavaScript cote client)

Controle d'acces

  • RBAC (Role-Based Access Control) : trois roles hierarchiques (admin, user, trusted) avec decorateurs Python verifiant les permissions a chaque requete
  • Protection CSRF : jeton anti-falsification inclus dans chaque formulaire POST, verifie cote serveur par Flask-WTF
  • Verification de propriete : chaque acces a une ressource verifie que l'utilisateur en est le proprietaire ou dispose d'une permission valide et non expiree

Chiffrement des documents

  • AES via Fernet (cryptography) : chiffrement optionnel des documents sensibles avec l'algorithme AES en mode CBC, encapsule par le schema Fernet qui garantit confidentialite et integrite
  • Activation par document : le proprietaire choisit individuellement quels documents chiffrer
  • Cle de chiffrement : geree par la configuration serveur, isolee du code applicatif
  • Dechiffrement transparent : le fichier est dechiffre a la volee lors du telechargement par un utilisateur autorise

Securite des fichiers

  • Stockage UUID : chaque fichier uploade est renomme avec un UUID v4 aleatoire, le nom original n'apparait jamais sur le serveur de fichiers
  • Validation d'extension : seules les extensions autorisees sont acceptees (PDF, images, documents Office, texte)
  • Limite de taille : 16 Mo maximum par fichier, rejetee cote serveur avant ecriture sur disque
  • Repertoire non navigable : le dossier de stockage est configure pour interdire le listing et l'acces direct

Tracabilite et audit

  • 21 types d'actions journalisees : chaque interaction significative est enregistree dans la table logs
  • Informations capturees : identifiant utilisateur, type d'action, details JSON, adresse IP, User-Agent, horodatage
  • Consultation par l'administrateur : interface de filtrage et de recherche dans les journaux
  • Non repudiation : les logs ne peuvent pas etre modifies ou supprimes par un utilisateur standard
