1. Contexte et objectifs

FamiliDocs est un coffre-fort numérique familial conçu pour centraliser, organiser et sécuriser l'ensemble des documents administratifs d'un foyer. L'application répond à unbesoin concret et universel : offrir un espace unique, protégé et accessible ou chaque membre de la famille peut stocker, retrouver et partager ses documents importants.

Le projet s'inscrit dans le cadre du BTS SIO option SLAM, épreuve E5 (Conception et maintenance de solutions informatiques). Il constitue le projet principal du candidat et illustre la maîtrise complète du cycle de développement d'une application web, de la conception à lamise en production, incluant une version exécutable desktop.

Objectifs fonctionnels

  • Centralisation : regrouper tous les documents administratifs (impôts, santé, banque, logement, identité) dans un espace unique et structuré
  • Organisation : classer par catégories, sous-dossiers, tags et niveaux de confidentialité pour retrouver instantanement n'importe quel document
  • Sécurisation : protéger les données sensibles par chiffrement AES, hachage bcrypt, gestion stricte des sessions et contrôle d'accès granulaire
  • Partage contrôle : permettre le partage de documents entre membres de confiance (conjoint, enfants, aidants) avec des permissions fines et temporaires
  • Collaboration familiale : créer des espaces famille avec rôles hiérarchiques, liens d'invitation et chat intégré
  • Suivi des échéances : alerter les utilisateurs sur les documents expirants, les tâches en cours et les deadlines administratives
  • Conformité RGPD : permettre l'export complet des données personnelles de chaque utilisateur

Public cible

  • Parents âgés: personnes peu à l'aise avec le numérique, nécessitant une interface simple et intuitive
  • Familles : foyers souhaitant centraliser les documents de tous les membres
  • Aidants familiaux : personnes accompagnant un proche dans ses démarches administratives
  • Particuliers : toute personne souhaitant sécuriser et organiser ses papiers importants

L'interface a étépensée pour êtreaccessible, claire et non intimidante, avec une navigation intuitive, des libellés explicites, un mode sombre et une compatibilité mobile.

2. Architecture technique

Pattern MVC adapté a Flask

L'application suit le pattern Model-View-Controller adapté aux conventions Flask :

  • Models : 13 classes SQLAlchemy déclaratives représentant les tables de la base de données PostgreSQL, avec relations, validations et méthodes métier
  • Views : 47 templates Jinja2 organisés par module fonctionnel, héritant d'un layout de base avec navigation dynamique et macros réutilisables
  • Controllers : 10 Blueprints Flask regroupant les routes par domaine (auth, user, documents, tasks, families, admin, notifications, search, versions, messages)
  • Services : 8 services métier pour le chiffrement, les fichiers, les notifications, les permissions, la recherche, les sauvegardes, le planificateur et l'authentification

4. Fonctionnalités détaillées

4.a Gestion des utilisateurs

Le système gère trois niveaux de rôles globaux avec des permissions hiérarchiques :

RôleDroitsUsage type
adminAccès total : gestion des utilisateurs, logs système, sauvegardes, configuration globaleAdministrateur du foyer
userCRUD sur ses propres documents, dossiers et tâches. Partage vers d'autres utilisateurs. Creation et gestion de famillesMembre standard de la famille
trustedDroits identiques a user + accès aux documents partagés avec permissions étenduesPersonne de confiance, aidant

Sécurité de connexion :

  • Rate limiting : maximum 5 tentatives echouees par adresse IP
  • Blocage automatique pendant 15 minutes après dépassement
  • Journalisation de chaque tentative (réussie ou echouee) avec IP et User-Agent
  • Politique de mot de passe stricte : 8 caracteres minimum, au moins 1 majuscule, 1 minuscule, 1 chiffre et 1 caractere special
  • Message de bienvenue automatique à l'inscription

4.b Gestion documentaire

Coeur fonctionnel de l'application, la gestion des documents couvre l'ensemble du cycle de vie des fichiers administratifs.

Formats supportes :

  • Documents : PDF, DOC, DOCX (Word), XLS, XLSX (Excel), TXT
  • Images : JPG, JPEG, PNG, GIF, BMP, WEBP
  • Limite de taille : 16 Mo par fichier

Dossiers hiérarchiques : classement par catégories prédéfinies à lacréation du compte :

  • Administratif : pieces d'identité, impôts, actes officiels
  • Santé : ordonnances, résultats d'analyses, carnet de santé
  • Banque : releves, contrats, justificatifs
  • Logement : bail, quittances, diagnostics, assurances
  • Autres : tous les documents ne relevant pas des catégories precedentes

L'utilisateur peut créer des sous-dossiers à l'intérieur de chaque catégorie pour affiner son classement.

Niveaux de confidentialité :

  • public : visible par tous les membres du foyer
  • private : visible uniquement par le propriétaire — chiffrement AES automatique
  • restricted : visible uniquement par les utilisateurs ayant recu une permission explicite

Operations en masse : selection de plusieurs documents avec checkboxes pour supprimer, deplacer ou changer la confidentialité en une seule action.

Barre de progression : indicateur visuel lors de l'upload de fichiers volumineux.

4.c Versioning des documents

Chaque document dispose d'un historique complet de ses versions, permettant la traçabilité et la récupération de versions antérieures.

  • Upload de nouvelle version : ajout d'un fichier actualise avec commentaire obligatoire
  • Historique chronologique : consultation de toutes les versions avec date, auteur et commentaire
  • Restauration : possibilité de restaurer n'importe quelle version antérieure comme version courante
  • Téléchargement individuel : chaque version peut êtretelechargee independamment
  • Numérotation automatique : numéro de version incrémental géré par le système

4.d Tags et recherche avancée

Système de tags :

  • Creation de tags personnalises avec couleur hexadecimale au choix
  • Association multiple : un document peut porter plusieurs tags, un tag peut êtreapplique a plusieurs documents
  • Filtrage rapide par tag depuis l'interface de listing
  • Gestion complète (création, modification, suppression) avec vérification d'intégrité

Recherche multi-critères :

  • Texte libre : recherche dans le nom du document et le nom original du fichier
  • Type de fichier : filtrage par extension (PDF, images, Word, Excel...)
  • Dossier : filtrage par catégorie ou sous-dossier
  • Tags : filtrage par un ou plusieurs tags
  • Dates : plage de dates de création ou de modification
  • Confidentialite : filtrage par niveau (public, prive, restreint)
  • Tri configurable : par nom, date, taille, type (ascendant ou descendant) avec colonnes cliquables
  • Pagination : 20 résultats par page avec navigation

Recherche globale AJAX : barre de recherche dans la sidebar avec suggestions en temps réel via dropdown, sans rechargement de page.

4.e Partage et permissions

Le système de partage permet un contrôle fin des droits accordés à chaqueutilisateur sur chaque document.

Permissions granulaires :

PermissionDescription
can_viewConsultation du document et de ses metadonnées
can_editModification des metadonnées (nom, tags, dossier)
can_downloadTéléchargement du fichier sur le poste local
can_shareRe-partage du document vers d'autres utilisateurs
  • Accès temporaire : chaque permission peut être assortie d'une date de fin (end_date) après laquelle l'accès est automatiquement révoqué
  • Révocation immediate : le propriétaire peut supprimer instantanement toute permission accordee
  • Notifications automatiques : le beneficiaire recoit une notification lors de l'attribution, la modification ou la revocation d'une permission
  • Traçabilité : chaque opération de partage est journalisée dans les logs

4.f Familles et collaboration

Fonctionnalité majeure permettant de créer des espaces de collaboration familiale avec rôles hiérarchiques.

Gestion des familles :

  • Creation d'une famille avec nom et description
  • Le createur obtient automatiquement le role chef_famille
  • Invitation de nouveaux membres par liens de partage sécurisés (token unique, expiration, limité d'utilisations)
  • Gestion des rôles : le chef de famille peut promouvoir ou rétrograder les membres (8 rôles disponibles)
  • Exclusion de membres par les chefs de famille ou administrateurs
  • Un utilisateur peut appartenir a plusieurs familles simultanement

Liens d'invitation :

  • Token aléatoire unique généré pour chaque lien
  • Date d'expiration configurable (par defaut 7 jours)
  • Nombre maximum d'utilisations configurable
  • Possibilite de révoquer un lien à toutmoment
  • Nettoyage automatique des liens expires

4.g Chat familial

Système de messagerie intégré à chaqueespace familial.

  • Messages en temps réel : envoi, édition et suppression de messages dans le chat de la famille
  • Annonces : les chefs de famille peuvent epingler des messages importants comme annonces
  • Edition et suppression : un auteur peut modifier ou supprimer ses propres messages (soft delete)
  • Chargement progressif : bouton "Charger plus" pour consulter l'historique des messages
  • Permissions : seuls les membres de la famille ont accès au chat

4.h Taches et échéances

Module de gestion des tâches administratives, liées ou non à undocument ou une famille.

4 niveaux de priorité :

PrioriteCouleurUsage
lowGrisTache sans urgence
mediumBleuTache standard
highOrangeTache importante
urgentRougeAction requise immediatement

4 statuts de progression : pending, in_progress, complèted, cancelled.

  • Assignation : possibilité d'assigner une tâchea un autre membre de la famille, avec notification automatique
  • Rappels configurables : date de rappel independante de la date d'échéance
  • Vue calendrier : visualisation des tâches sur un calendrier mensuel
  • Vue tâches en retard : page dediee listant les tâches depassees
  • Vue tâches a venir : page dediee listant les tâches des 7 prochains jours
  • Confirmation de suppression : modal de confirmation avant suppression d'une tache
  • Pagination : 20 tâches par page avec tri par colonnes

4.i Notifications

Système de notifications complet avec 12 types assurant la communication en temps réel.

  • task_due / task_overdue / task_assigned : échéances et assignation de tâches
  • document_expiry / document_expired / document_shared : cycle de vie des documents
  • permission_granted / permission_revoked / permission_expiring : droits d'accès
  • system / backup_complète / welcome : notifications système

Comportement :

  • Badge temps réel dans la barre de navigation avec compteur de notifications non lues
  • Widget resume sur le dashboard (non-lus, urgents, recents)
  • Rafraîchissement automatique toutes les 60 secondes via requête AJAX
  • Marquage lu/non lu individuel ou en masse
  • Suppression en masse des notifications lues
  • Expiration automatique configurable
  • 4 niveaux de priorité : low, normal, high, urgent

4.j Tableau de bord

Page d'accueil personnalisée offrant une vue synthétique de l'activité de l'utilisateur.

  • 4 compteurs : documents, dossiers, tâches en cours, documents partagés
  • Repartition par type de fichier : graphique (PDF, images, Word, Excel, autres)
  • Repartition des tâches par statut : diagramme pending / in_progress / complèted / cancelled
  • Graphe d'activité 6 mois : courbe d'evolution de l'activité

5. Sécurité

La sécurité constitue un pilier central de FamiliDocs, l'application gérant des documents administratifs sensibles. Chaque couche de l'architecture intègre des mécanismes de protection spécifiques.

Authentification et mots de passe

  • Hachage bcrypt : chaque mot de passe est hache avec un salt aléatoire unique et un cout de 12 rounds, rendant les attaques par force brute et par tables arc-en-ciel impraticables
  • Politique de mot de passe : minimum 8 caracteres, au moins 1 majuscule, 1 minuscule, 1 chiffre et 1 caractere special
  • Rate limiting : maximum 5 tentatives de connexion echouees par adresse IP, suivi d'un blocage automatique de 15 minutes
  • Sessions sécurisées : durée de vie de 2 heures, cookie HttpOnly (inaccessible au JavaScript côté client)
  • Warning d'expiration : alerte JavaScript 5 minutes avant l'expiration de la session

Contrôle d'accès

  • RBAC (Rôle-Based Access Control) : 3 rôles globaux (admin, user, trusted) + 8 rôles familiaux, avec décorateurs Python vérifiant les permissions à chaquerequête
  • Protection CSRF : jeton anti-falsification inclus dans chaque formulaire POST, vérifié côté serveur par Flask-WTF
  • Verification de propriété : chaque accès à uneressource vérifié que l'utilisateur en est le propriétaire ou dispose d'une permission valide et non expirée
  • Decorateurs : @login_required sur toutes les routes protégées, @admin_required sur les routes d'administration

Chiffrement des documents

  • AES via Fernet (cryptography) : chiffrement des documents sensibles avec l'algorithme AES en mode CBC, encapsule par le schema Fernet qui garantit confidentialité et intégrité
  • Chiffrement automatique : les documents marqués private sont automatiquement chiffrésa l'upload
  • Cle de chiffrement : gèree par la configuration serveur (ENCRYPTION_KEY dans .env), isolée du code applicatif
  • Déchiffrement transparent : le fichier est déchiffré à lavolée lors du téléchargement par un utilisateur autorisé

Sécurité des fichiers

  • Stockage UUID : chaque fichier uploadé est renomme avec un UUID v4 aléatoire, le nom original n'apparait jamais sur le serveur
  • Validation d'extension : seules les extensions autorisées sont acceptées (PDF, images, documents Office, texte)
  • Limite de taille : 16 Mo maximum par fichier, rejetee côté serveur avant ecriture sur disque
  • Répertoire non navigable : le dossier de stockage est configuré pour interdire le listing et l'accès direct

En-tetes HTTP de sécurité

Traçabilité et audit

  • 22 types d'actions journalisées : chaque interaction significative est enregistree dans la table logs
  • Informations capturees : identifiant utilisateur, type d'action, details JSON, adresse IP, User-Agent, horodatage
  • Consultation par l'administrateur : interface de filtrage et de recherche dans les journaux avec pagination
  • Non repudiation : les logs ne peuvent pas êtremodifies ou supprimes par un utilisateur standard

8. Déploiement et exécutable

Version web

  • Prerequis : Python 3.12+, PostgreSQL 16, pip
  • Installation : pip install -r requirements.txt (36 dependances)
  • Configuration : copier .env.example en .env et renseigner DATABASE_URL, SECRET_KEY, ENCRYPTION_KEY
  • Lancement : python app/main.py → accessible sur http://localhost:5000

Version desktop exécutable (.exe)

FamiliDocs peut êtrecompilé en exécutable Windows autonome grace a PyInstaller.

  • Script : build_exe.bat automatise la création de l'exécutable
  • Configuration : familidocs.spec liste tous les hidden imports (modèles, services, cryptography)
  • Deux lanceurs :
    • desktop_app.py : fenetre native CustomTkinter avec webview intégré
    • desktop_launcher.py : ouvre le navigateur par defaut
  • BDD et fichiers 100% partagés : la version web et le .exe utilisent exactement la meme base PostgreSQL et le meme dossier app/database/ (uploads, backups, avatars). Aucune duplication de données
  • Dossier unique : tout est centralise dans PycharmProjects/FamiliDocs/app/database/ — pas de dossier externe ~/.familidocs en usage normal
  • Detection automatique : gétattr(sys, 'frozen', False) detecte le mode PyInstaller. Si le .exe est compilé sans PostgreSQL disponible, fallback SQLite local

Dossier de données unique — app/database/

FamiliDocs/app/database/
├── uploads/           # Fichiers uploadés (web + exe)
│   └── avatars/       # Photos de profil
├── backups/           # Sauvegardes

La version web (python app/main.py) et la version desktop (desktop_launcher.py ou .exe) partagent ce meme répertoire. Un document uploadé via le web est immediatement visible depuis le .exe, et inversement.
