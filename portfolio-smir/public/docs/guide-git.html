1. Introduction à Git

Le contrôle de version est un système qui enregistre les modifications apportées à un ensemble de fichiers au fil du temps. Il permet de suivre l'historique complet, de travailler à plusieurs sans conflit et de revenir à un état antérieur en cas de problème.

Centralisé (SVN, CVS)

  • Un seul serveur contient tout l'historique
  • Connexion permanente nécessaire
  • Point de défaillance unique

Distribué (Git, Mercurial)

  • Chaque développeur possède une copie complète
  • Travail hors ligne possible
  • Branches légères et rapides

Git a été créé en 2005 par Linus Torvalds (créateur de Linux) avec des objectifs clairs : vitesse, conception simple, support du développement non-linéaire et capacité à gérer de très gros projets. Aujourd'hui, plus de 95% des développeurs l'utilisent.

3. Concepts fondamentaux

Les trois zones de Git

ZoneDescriptionCommande
Working DirectoryRépertoire de travail — les fichiers tels que vous les éditezVotre éditeur
Staging Area (Index)Zone intermédiaire — prépare le prochain commitgit add
Repository (.git)Base de données locale contenant tout l'historiquegit commit

Les 4 états d'un fichier

  • Untracked — Nouveau fichier, non suivi par Git
  • Unmodified — Fichier suivi, identique au dernier commit
  • Modified — Fichier modifié depuis le dernier commit
  • Staged — Modification ajoutée à la staging area, prête pour le commit

Structure du répertoire .git

Structure interne
.git/
  ├── HEAD            # Pointe vers la branche actuelle
  ├── config          # Configuration locale du depot
  ├── hooks/          # Scripts executes automatiquement
  ├── index           # La staging area (binaire)
  ├── objects/        # Objets Git (blobs, trees, commits)
  ├── refs/           # References (branches, tags, remotes)
  └── logs/           # Historique des references
Attention

Ne modifiez jamais manuellement le contenu du répertoire .git. Toute manipulation incorrecte peut corrompre l'historique de manière irréversible.

Annuler des modifications

CommandeEffetDanger

Attention — git reset --hard

Cette commande supprime définitivement les modifications non commitées. Privilégiez git stash pour sauvegarder temporairement.

5. Branches

Les branches permettent de travailler sur plusieurs fonctionnalités en parallèle sans affecter la branche principale.

Bash — Gestion des branches
git branch                          # Lister les branches locales
git branch -a                       # Toutes les branches (locales + distantes)
git branch feature/auth             # Creer une branche
git switch feature/auth             # Basculer sur une branche
git switch -c feature/panier        # Creer + basculer
git branch -d feature/auth          # Supprimer (branche fusionnee)
git branch -D feature/exp           # Forcer la suppression
git branch -m nouveau-nom           # Renommer la branche courante

Conventions de nommage

PréfixeUsageExemple
feature/Nouvelle fonctionnalitéfeature/panier-achat
bugfix/Correction de bugbugfix/login-redirect
hotfix/Correction urgente en productionhotfix/faille-xss
release/Préparation d'une versionrelease/2.1.0
chore/Tâches techniqueschore/update-deps
Tip — Alias utile

Créez un alias pour visualiser vos branches : git config --global alias.lg "log --oneline --graph --all --decorate". Tapez ensuite git lg.

6. Merge et Rebase

Fast-forward merge

La branche cible n'a pas divergé : Git avance le pointeur. Aucun commit de merge.

Bash
git switch main
git merge feature/auth

Merge 3-way

Les branches ont divergé : Git crée un commit de merge avec deux parents.

Bash
git switch main
git merge feature/panier

Résoudre un conflit

Fichier en conflit
function getUser() {
<<<<<<< HEAD
  return fetchUserFromAPI('/api/v2/users');
=======
  return fetchUserFromCache();
>>>>>>> feature/cache
}
Bash — Résolution
# 1. Ouvrir le fichier, choisir la bonne version, supprimer les marqueurs
# 2. Marquer comme resolu
git add fichier-en-conflit.js
# 3. Finaliser le merge
git commit

Rebase

Bash
# Rebaser feature/auth sur main (historique lineaire)
git switch feature/auth
git rebase main

# Puis fast-forward merge
git switch main
git merge feature/auth

Merge vs Rebase

CritèreMergeRebase
HistoriqueNon-linéaire, chronologie réelleLinéaire et propre
Commit supplémentaireOui (merge commit)Non
SécuritéNe réécrit pas l'historiqueRéécrit les SHA
Branches partagéesSûrDangereux
UsageIntégration dans mainMise à jour d'une branche locale
Règle d'or du Rebase

Ne jamais rebaser une branche pushée et partagée. Le rebase réécrit les SHA, causant des conflits pour les collaborateurs.

Rebase interactif

Bash
# Reorganiser les 4 derniers commits
git rebase -i HEAD~4

# Actions : pick (garder), squash (fusionner),
#           reword (modifier message), drop (supprimer)

7. GitHub

Bash — Remotes
git remote -v                                         # Lister les remotes
git remote add origin git@github.com:user/repo.git    # Ajouter
git remote set-url origin git@github.com:user/new.git # Modifier l'URL

Synchronisation

CommandeDirectionDescription
git pushLocal → DistantEnvoyer les commits locaux vers le remote
git pullDistant → LocalRécupérer et fusionner (fetch + merge)
git fetchDistant → LocalRécupérer sans fusionner
Bash
git push -u origin feature/auth     # Push + tracking (1ere fois)
git push origin main                # Push classique
git fetch --all --prune             # Fetch + nettoyage branches supprimees
git pull origin main                # Pull (fetch + merge)

Fork vs Clone

Clone

  • Copie locale d'un dépôt
  • Pour les collaborateurs directs
  • Push autorisé si droits d'accès

Fork

  • Copie serveur sur votre compte GitHub
  • Pour contribuer à l'open source
  • Contribution via Pull Request
Tip — Workflow Fork

Après un fork, ajoutez le dépôt original comme remote upstream : git remote add upstream https://github.com/original/repo.git puis git pull upstream main régulièrement.

8. Pull Requests

Une Pull Request (PR) est une demande de fusion d'une branche vers une autre. C'est le mécanisme central de la revue de code en équipe.

Bash — Workflow PR
# 1. Creer et travailler sur une branche
git switch -c feature/dashboard
git add . && git commit -m "feat: ajout page dashboard"

# 2. Pousser la branche
git push -u origin feature/dashboard

# 3. Creer la PR via GitHub CLI
gh pr create --title "feat: ajout dashboard" \
  --body "Ajout de la page dashboard avec graphiques."

Processus de revue

  1. Ouverture — Créer la PR avec titre clair et description
  2. Revue — Les reviewers commentent et suggèrent des modifications
  3. Corrections — Le développeur corrige et re-push
  4. Approbation — Au moins un reviewer approuve
  5. Merge — Fusion dans la branche cible

Stratégies de merge

StratégieRésultatQuand l'utiliser
Merge commitConserve tous les commits + merge commitHistorique détaillé souhaité
Squash and mergeUn seul commit regroupéBeaucoup de petits commits
Rebase and mergeCommits re-appliqués (linéaire)Historique propre
Bonnes pratiques PR

Gardez les PR petites (moins de 400 lignes). Une PR volumineuse est difficile à relire et ralentit le développement. Découpez les grosses fonctionnalités en plusieurs PR successives.

10. Workflow professionnel

Git Flow

Modèle complet pour les projets avec cycles de release définis :

  • main — Code en production, toujours stable
  • develop — Branche d'intégration
  • feature/* — Fonctionnalités (depuis develop)
  • release/* — Préparation de version
  • hotfix/* — Corrections urgentes (depuis main)

GitHub Flow

Workflow simplifié pour le déploiement continu :

  1. Créer une branche depuis main
  2. Développer et commiter
  3. Ouvrir une Pull Request
  4. Revue de code et approbation
  5. Merger dans main et déployer

Trunk-Based Development

Approche minimaliste (Google, Meta) : tout le monde travaille sur main avec des branches à très courte durée, des commits fréquents et des feature flags. CI/CD obligatoire.

CritèreGit FlowGitHub FlowTrunk-Based
ComplexitéÉlevéeModéréeFaible
Branches5+ permanentesmain + featuremain surtout
DéploiementPar releaseContinu (jours)Continu (heures)
ÉquipeGrandePetite à moyenneToute taille
IdéalApps versionnéesWeb / SaaSDevOps mature
Recommandation

Pour un projet BTS SIO ou une équipe de 2 à 5 développeurs, GitHub Flow est le meilleur choix : simple, basé sur les PR, compatible avec le déploiement continu.

Commits atomiques

À éviter

git add .
git commit -m "modifs diverses"

Un commit mélangeant feature, fix et style.

À préférer

git add src/auth/
git commit -m "feat(auth): login"

git add src/header.css
git commit -m "style: header"

Un commit = un sujet précis.

Attention — Secrets

Ne commitez jamais de mots de passe, clés API ou fichiers .env. Même supprimés ensuite, ils restent dans l'historique Git.

Tags et releases

Bash
git tag -a v1.0.0 -m "Version 1.0.0 - release stable"   # Tag annote
git tag --list                                             # Lister les tags
git push origin v1.0.0                                     # Pousser un tag
git push origin --tags                                     # Pousser tous les tags

Git Stash

Bash
git stash                             # Sauvegarder les modifs en cours
git stash push -m "WIP: formulaire"   # Avec message descriptif
git stash list                        # Lister les stashes
git stash pop                         # Restaurer + supprimer
git stash apply stash@{2}             # Restaurer sans supprimer
git stash drop stash@{0}              # Supprimer un stash

12. GitHub Actions (CI/CD)

GitHub Actions exécute automatiquement des tâches (tests, lint, build, déploiement) en réponse à des événements sur le dépôt.

TermeDescription
WorkflowProcessus défini en YAML dans .github/workflows/
EventDéclencheur (push, pull_request, schedule)
JobEnsemble d'étapes sur un même runner
StepAction individuelle au sein d'un job
RunnerMachine virtuelle (ubuntu-latest, windows-latest)

Exemple — Pipeline CI (Lint + Tests)

YAML — .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20]

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'

      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build

Badge de statut

Markdown — README.md
![CI](https://github.com/user/repo/actions/workflows/ci.yml/badge.svg)

Pipeline type pour une application web

  1. Push sur feature/* → Lint + tests unitaires
  2. Pull Request vers main → Lint + tests + build + preview
  3. Merge dans main → Lint + tests + build + déploiement production
YAML — .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.ORG_ID }}
          vercel-project-id: ${{ secrets.PROJECT_ID }}
          vercel-args: '--prod'
Tip — Secrets GitHub

Les tokens et clés API utilisés dans les workflows doivent être stockés dans Settings > Secrets and variables > Actions. Ils sont chiffrés et jamais exposés dans les logs.
