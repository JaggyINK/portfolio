1. Contexte

Annonz est une plateforme d'annonces full-stack developpee dans le cadre du BTS SIO. Il s'agit d'une application web complete permettant aux utilisateurs de publier, rechercher et gerer des annonces en ligne.

Le projet met en oeuvre une architecture MVC manuelle en PHP natif, sans framework, afin de maitriser les fondamentaux du developpement web backend et de comprendre les mecanismes sous-jacents habituellement abstraits par les frameworks.

2. Architecture technique

Backend

  • PHP 8.x natif (sans framework) pour une maitrise complete du code
  • Architecture MVC manuelle avec separation stricte des responsabilites
  • Routing personnalise via .htaccess et un front controller
  • PDO pour l'acces aux donnees avec prepared statements

Base de donnees

  • MySQL avec moteur InnoDB pour les transactions et l'integrite referentielle
  • Index FULLTEXT pour la recherche avancee par mots-cles
  • Relations et contraintes de cles etrangeres

Frontend

  • HTML5 semantique pour la structure des pages
  • CSS3 avec design responsive (media queries, flexbox, grid)
  • JavaScript vanilla pour les interactions dynamiques (validation, AJAX)

Serveur

  • Compatible Apache (avec mod_rewrite) et Nginx
  • Configuration de reecriture d'URL pour des routes propres

3. Fonctionnalites

a. Gestion des annonces

  • CRUD complet — Creation, lecture, modification et suppression d'annonces
  • Attribution de categories pour organiser les annonces
  • Upload d'images pour illustrer chaque annonce
  • Statut de l'annonce (active, en attente, archivee)
  • Pagination des resultats pour une navigation fluide

b. Recherche full-text

  • Index MySQL FULLTEXT sur les colonnes titre et description
  • Recherche par mots-cles avec classement par pertinence (MATCH ... AGAINST)
  • Filtrage par categorie, fourchette de prix et localisation
  • Tri des resultats par pertinence, date de publication ou prix
  • Suggestions de recherche basees sur les termes populaires

c. Authentification

  • Inscription avec validation des champs (email, mot de passe fort)
  • Connexion securisee avec gestion de sessions PHP
  • Hashage des mots de passe avec bcrypt via password_hash()
  • Validation de l'adresse email lors de l'inscription
  • Gestion des roles (utilisateur, administrateur)
  • Deconnexion avec destruction complete de la session

d. Upload securise

  • Validation du type MIME reel du fichier (pas seulement l'extension)
  • Limite de taille configurable (ex. 2 Mo maximum par image)
  • Renommage securise avec uniqid() pour eviter les collisions et injections
  • Stockage organise dans des repertoires structures par date
  • Whitelist d'extensions autorisees (jpg, jpeg, png, webp)

e. Moderation

  • Systeme de signalement d'annonces par les utilisateurs
  • Interface d'administration pour la validation des annonces signalees
  • Gestion des utilisateurs (suspension, suppression de compte)
  • Tableau de bord administrateur avec statistiques

f. Messagerie

  • Systeme de contact entre utilisateurs via l'annonce
  • Formulaire de contact lie a chaque annonce
  • Boite de reception pour consulter les messages recus
  • Notification de nouveaux messages

5. Securite

Protection contre l'injection SQL

Toutes les requetes sont executees via des prepared statements PDO. Les parametres utilisateurs ne sont jamais concatenes directement dans les requetes SQL, eliminant ainsi le risque d'injection.

Protection XSS

  • Echappement systematique des sorties avec htmlspecialchars()
  • En-tetes Content-Security-Policy (CSP) pour limiter les sources de scripts
  • Validation et sanitization des entrees utilisateur

Protection CSRF

  • Generation de tokens CSRF uniques par session
  • Verification du token sur chaque formulaire soumis en POST
  • Renouvellement du token apres chaque action sensible

Hashage des mots de passe

  • Utilisation de password_hash() avec l'algorithme bcrypt
  • Verification via password_verify() sans jamais comparer en clair
  • Cout de hashage configure pour un bon equilibre securite/performance

Validation cote serveur

  • Toutes les entrees utilisateur sont validees cote serveur (jamais uniquement cote client)
  • Verification des types, longueurs, formats et plages de valeurs
  • Messages d'erreur generiques pour ne pas divulguer d'informations sensibles

Upload securise

  • Whitelist stricte d'extensions autorisees (.jpg, .jpeg, .png, .webp)
  • Verification du type MIME reel avec finfo_file()
  • Stockage en dehors du repertoire web accessible quand possible

6. Architecture MVC

L'architecture Modele-Vue-Controleur (MVC) permet de separer clairement les responsabilites au sein de l'application, facilitant la maintenance et l'evolution du code.

Models — Acces aux donnees

  • Classes PHP dediees a l'interaction avec la base de donnees via PDO
  • Un modele par entite (UserModel, AnnonceModel, CategoryModel, MessageModel)
  • Methodes CRUD standardisees (find, findAll, create, update, delete)
  • Encapsulation de la logique d'acces aux donnees

Views — Templates HTML/PHP

  • Fichiers PHP melangeant HTML et donnees transmises par le controleur
  • Systeme de layout avec header/footer partages
  • Echappement systematique des variables affichees
  • Templates reutilisables pour les composants recurrents (cards, formulaires)

Controllers — Logique metier et routing

  • Front controller (index.php) recevant toutes les requetes
  • Routeur analysant l'URL pour dispatcher vers le bon controleur/methode
  • Chaque controleur gere un domaine fonctionnel (AuthController, AnnonceController, AdminController)
  • Validation des entrees, appel aux modeles, transmission des donnees aux vues
  • Middlewares pour la verification d'authentification et d'autorisation
