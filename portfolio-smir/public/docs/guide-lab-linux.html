1. Objectif du laboratoire

Ce laboratoire virtualisé a pour but de fournir un environnement isolé et reproductible pour la pratique de la cybersécurité, l'administration système Linux et le développement web. Il repose sur trois machines virtuelles, chacune ayant un rôle distinct :

VM Rôle Description
Kali Linux Attaquant Distribution offensive pour les tests d'intrusion, scans et exploitation
Ubuntu Services Serveur de services (web, bases de données, DNS, etc.)
Lubuntu Cible vulnérable Machine légère hébergeant des applications volontairement vulnérables

Cas d'usage concrets

  • Simulation réseau : mise en place d'un LAN virtualisé pour tester les communications inter-VM
  • Attaques contrôlées : injections SQL (SQLi), brute force, scans de ports (Nmap), exploitation de failles
  • Administration Linux : installation, configuration, optimisation et sécurisation de systèmes
  • Déploiement web : serveurs Apache/Nginx, PHP, MariaDB sur Ubuntu et Lubuntu
Conseil

Ce lab est conçu pour un usage strictement pédagogique et éthique. N'utilisez jamais ces techniques sur des systèmes sans autorisation explicite.

2. Prérequis matériels et logiciels

Configuration de la machine hôte

Composant Minimum Recommandé
Système d'exploitation Windows 10 Windows 11
Processeur 4 coeurs 6–8 coeurs
RAM 16 GB 16 GB+
Stockage SSD 150 GB libre SSD 200 GB+
Virtualisation VT-x (Intel) ou AMD-V activé dans le BIOS

Hyperviseur

VMware Workstation Pro (ou Player) est utilisé comme hyperviseur. Il offre une gestion avancée des snapshots, du réseau virtuel et des VMware Tools pour une intégration optimale avec l'hôte.

Attention

Assurez-vous que la virtualisation (VT-x / AMD-V) est bien activée dans les paramètres BIOS/UEFI de votre machine avant d'installer VMware. Sans cela, les VM ne pourront pas démarrer.

4. Création d'une VM type (VMware)

Procédure de base

  1. Ouvrir VMware → Create a New Virtual Machine
  2. Choisir Typical (recommended)
  3. Sélectionner I will install the operating system later
  4. Guest OS : Linux → Version correspondante (Debian 11+, Ubuntu 64-bit)
  5. Nommer la VM et choisir l'emplacement de stockage
  6. Configurer le disque : Store virtual disk as a single file
  7. Cliquer sur Customize Hardware pour ajuster CPU, RAM et réseau

Paramètres recommandés par VM

Paramètre Kali Linux Ubuntu Lubuntu
vCPU 2–4 2–4 1–2
RAM 4 GB 4–8 GB 2 GB
Disque 40–60 GB 60–80 GB 20–40 GB
Réseau Bridged Bridged Bridged
Stockage disque Single file Single file Single file

5. Configuration réseau

Mode Bridged (recommandé)

En mode Bridged, chaque VM obtient une adresse IP directement sur le LAN physique de l'hôte, comme si elle était une machine réelle connectée au même réseau. Cela permet une communication transparente entre les VM et avec les autres machines du réseau local.

Plan d'adressage IP

Machine Adresse IP Rôle
Kali Linux 192.168.200.140 Attaquant
Lubuntu 192.168.200.130 Cible vulnérable
Ubuntu 192.168.200.150 Serveur de services
Passerelle (Gateway) 192.168.200.1 Routeur / box

Alternative : mode Host-Only

Pour un lab complètement isolé du réseau local (aucun accès internet, aucune visibilité externe), utilisez le mode Host-Only. Cela crée un réseau privé entre l'hôte et les VM uniquement. Idéal pour les tests offensifs agressifs sans risque de fuite réseau.

Attention

En mode Bridged, vos VM sont visibles sur le réseau local. Ne lancez jamais d'attaques depuis Kali vers des machines qui ne vous appartiennent pas.

6. Installation de Kali Linux

Procédure d'installation

  1. Monter l'ISO Kali dans le lecteur CD/DVD de la VM
  2. Démarrer la VM et choisir Graphical Install
  3. Langue : Français
  4. Nom d'hôte : kali-lab
  5. Créer un utilisateur et définir le mot de passe
  6. Partitionnement : Assisté - utiliser un disque entier (ext4)
  7. Installer le chargeur d'amorçage GRUB sur le disque principal
  8. Redémarrer après l'installation

Post-installation

bash# Mise a jour complete du systeme
sudo apt update && sudo apt full-upgrade -y

# Installation des VMware Tools pour l'integration hote
sudo apt install -y open-vm-tools-desktop

# Redemarrage
sudo reboot
Conseil

Après le redémarrage, créez immédiatement un snapshot nommé « Kali – propre » dans VMware. Ce point de restauration servira de base saine pour tous vos futurs tests.

Procédure d'installation

  1. Monter l'ISO Ubuntu dans le lecteur CD/DVD de la VM
  2. Démarrer la VM et choisir Install Ubuntu
  3. Langue : Français
  4. Type d'installation : Installation normale (inclut les utilitaires et le navigateur)

Conseil

Séparer /home de / permet de réinstaller le système sans perdre les données utilisateur. C'est une bonne pratique en environnement de lab comme en production.

8. Installation de Lubuntu

Lubuntu est une distribution ultra-légère basée sur Ubuntu, utilisant l'environnement de bureau LXQt. Son faible besoin en ressources en fait le choix idéal pour une cible vulnérable dans le lab.

Procédure d'installation

  1. Monter l'ISO Lubuntu dans le lecteur CD/DVD de la VM
  2. Démarrer la VM et choisir Install Lubuntu
  3. Langue : Français
  4. Partitionnement : Assisté - utiliser un disque entier
  5. Finaliser l'installation et redémarrer

Déploiement de la stack vulnérable

bash# Mise a jour du systeme
sudo apt update

# Installation de la stack LAMP
sudo apt install -y apache2 php php-mysqli mariadb-server

# Verifier que les services sont actifs
sudo systemctl status apache2
sudo systemctl status mariadb
bash# Deployer l'application vulnerable dans le repertoire web
sudo cp -r /chemin/vers/vulnerable_app /var/www/html/vulnerable_app

# Definir les permissions
sudo chown -R www-data:www-data /var/www/html/vulnerable_app
sudo chmod -R 755 /var/www/html/vulnerable_app
Conseil

Vous pouvez utiliser des projets comme DVWA (Damn Vulnerable Web Application) ou bWAPP comme application vulnérable de référence pour vos tests d'intrusion.

9. Configuration réseau (Netplan)

Sur les distributions basées sur Ubuntu (y compris Lubuntu), la configuration réseau statique se fait via Netplan. Le fichier de configuration se trouve dans /etc/netplan/.

Exemple : IP statique pour Lubuntu (192.168.200.130)

yaml# /etc/netplan/01-netcfg.yaml
network:
  version: 2
  renderer: networkd
  ethernets:
    ens33:
      dhcp4: no
      addresses:
        - 192.168.200.130/24
      routes:
        - to: default
          via: 192.168.200.1
      nameservers:
        addresses:
          - 1.1.1.1
          - 8.8.8.8

Appliquer la configuration

bash# Appliquer les modifications reseau
sudo netplan apply

# Verifier la nouvelle adresse IP
ip addr show ens33

# Tester la connectivite
ping -c 4 192.168.200.1
ping -c 4 1.1.1.1
Attention

L'indentation dans les fichiers YAML est critique (espaces uniquement, pas de tabulations). Une erreur d'indentation dans Netplan empêchera l'application de la configuration réseau.

Vérification de la connectivité inter-VM

bash# Depuis Kali (192.168.200.140), tester la connexion vers les autres VM :
ping -c 4 192.168.200.130   # Lubuntu
ping -c 4 192.168.200.150   # Ubuntu

# Scan rapide depuis Kali pour confirmer la visibilite reseau :
nmap -sn 192.168.200.0/24

10. Optimisation des ressources

Répartition sur un hôte 16 GB RAM

Machine RAM vCPU Note
Kali Linux 4 GB 2 Outils offensifs, interface graphique
Ubuntu 4 GB 2 Serveur de services
Lubuntu 2 GB 1 Cible légère
Hote Windows 4–6 GB — Restant pour l'OS hôte
Conseil

Ne démarrez pas les 3 VM simultanément si votre hôte n'a que 16 GB de RAM. Privilégiez le démarrage de 2 VM à la fois (ex. Kali + Lubuntu pour les tests offensifs).

Installation des VMware Tools

Les VMware Tools améliorent les performances (graphiques, copier-coller, dossiers partagés) :

bash# Installation sur toutes les VM basees Debian/Ubuntu
sudo apt install -y open-vm-tools open-vm-tools-desktop

# Redemarrer pour activer les outils
sudo reboot

Désactiver les services inutiles

Pour économiser des ressources sur les VM, désactivez les services qui ne sont pas nécessaires en environnement lab : 

bash# Desactiver Bluetooth (inutile en VM)
sudo systemctl disable bluetooth.service

# Desactiver le service d'impression CUPS
sudo systemctl disable cups.service

# Desactiver le service de decouverte reseau Avahi
sudo systemctl disable avahi-daemon.service

# Verifier les services desactives
sudo systemctl list-unit-files --state=disabled | grep -E "bluetooth|cups|avahi"

11. Snapshots et bonnes pratiques

Les snapshots VMware sont essentiels dans un lab de cybersécurité. Ils permettent de capturer l'état complet d'une VM à un instant T et d'y revenir instantanément après un test destructif.

Stratégie de snapshots recommandée

VM Nom du snapshot Quand le créer
Kali Kali – clean Après l'installation + mise a jour + VMware Tools
Lubuntu Lubuntu – vuln app installée Après déploiement de l'application vulnérable
Ubuntu Ubuntu – serveur OK Après configuration des services (Apache, MariaDB, etc.)

Bonnes pratiques

  • Avant chaque test agressif : créez un snapshot pour pouvoir restaurer la VM en cas de corruption
  • Nommez vos snapshots : utilisez des noms explicites (date + état) pour retrouver facilement un point de restauration
  • Nettoyez les snapshots obsolètes : chaque snapshot consomme de l'espace disque supplémentaire
  • Documentez vos tests : notez les manipulations effectuées entre chaque snapshot pour garder une trace
  • Sauvegardez les VM : exportez périodiquement vos VM en OVA pour avoir une copie de secours complète
Attention

Les snapshots ne remplacent pas une sauvegarde. Si le disque physique de l'hôte est endommagé, les snapshots seront perdus. Exportez régulièrement vos VM en fichiers OVA sur un support externe.

Commande rapide : créer un snapshot VMware en CLI

bash# Depuis l'hote Windows (vmrun doit etre dans le PATH)
vmrun -T ws snapshot "C:\VMs\Kali\Kali.vmx" "Kali - clean"

# Lister les snapshots existants
vmrun -T ws listSnapshots "C:\VMs\Kali\Kali.vmx"

# Restaurer un snapshot
vmrun -T ws revertToSnapshot "C:\VMs\Kali\Kali.vmx" "Kali - clean"
