1. Contexte et objectifs

L'Intranet CPMS V2 est un portail RH numérique interne conçu pour la Caisse de Prévoyance Mutuelle Santé (CPMS). L'application centralise l'ensemble des documents administratifs, outils RH et services internes dans une interface unique, moderne et accèssible a tous les collaborateurs.

Le projet s'inscrit dans le cadre du BTS SIO option SLAM, épreuve E5 (Conception et maintenance de solutions informatiques). Il constitue une refonte complète de l'intranet V1, migrant d'une architecture monolithique vers une architecture decouplée SPA + API REST, avec intégration de multiples systèmes d'entreprise (Active Directory, AS400, LDAPS).

Objectifs fonctionnels

  • Centralisation documentaire : regrouper tous les documents RH (mutuelle, prevoyance, chartes, conventions, aides au logement, epargne salariale) dans un espace unique avec visionneuse PDF integree
  • Self-service mot de passe : permettre aux collaborateurs de changer leur mot de passe sur trois systèmes simultanement (Active Directory, AS400, ODBC) sans solliciter le support informatique
  • Déblocage de compte : offrir un mécanisme autonome de déblocage de compte AD via questions de sécurité, avec contrôle anti-brute-force
  • Administration centralisée : fournir un tableau de bord aux administrateurs pour gérer les comptes verrouillés/suspendus, consulter les statistiques de déblocage et intervenir manuellement
  • Hub d'outils internes : referencer et documenter les 12+ outils et services utilises par l'entreprise (iLucca, Nibelis, Swile, Teale, Deskare, JIRA, Digiposte, etc.)
  • FAQ interactive : proposer un chatbot et un accordeon FAQ couvrant 50 questions reparties en 11 catégories pour reduire la charge du service RH
  • Experience utilisateur : offrir une interface responsive, un mode sombre, un système de favoris avec drag-and-drop, une meteo en temps réel et une recherche globale

Public cible

  • Collaborateurs CPMS : tous les salaries de l'entreprise, tout profil technique confondu
  • Service RH : équipe de 3 personnes utilisant le portail pour orienter les collaborateurs et publier des documents
  • Administrateurs IT : équipe technique utilisant le dashboard admin pour la gestion des comptes AD
  • Managers : encadrants consultant les tutoriels video de validation des absences/pointages

L'interface a ete conçue avec l'identite visuelle CPMS (palette teal/turquoise), la police Montserrat, un design responsive mobile/tablette/desktop et une navigation intuitive par cartes.

  • Frontend (SPA) : application Vue 3 single-page servie en statique, communiquant exclusivement par requêtes HTTP avec l'API
  • Backend (API) : API REST Laravel sans état, authentifiant via 3 mécanismes selon le contexte (Bearer admin AD, Bearer RH bcrypt, Bearer chat AD) plus CSRF cookie pour les routes publiques
  • Active Directory : connexion LDAPS (port 636) pour l'authentification admin, le module Chat, le changement de mot de passe AD et le déblocage de compte
  • MariaDB / MySQL : base relationnelle stockant les questions de sécurité, les comptes RH (bcrypt), les annonces/alertes/événements RH, les messages du chat, les médias, les tokens et l'audit
  • Middleware ODBC : script Python appelant l'AS400 via pyodbc pour les changements de mot de passe IBM i
  • Stockage de fichiers : dossier storage/app/hr-media exposé via lien symbolique storage:link pour les images/PDF uploadés depuis l'Espace RH

Utilitaire API — src/utils/api.js (52 lignes)

Deux fonctions centrales pour les appels authentifies :

  • authFetch(path, options) : construit l'URL complète depuis VITE_API_BASE, injecte le Bearer token depuis localStorage.admin_token, serialise les corps en JSON, desactive les cookies (credentials: 'omit'), force cache: 'no-store'
  • authJson(path, options) : appelle authFetch(), parse la réponse JSON, effectue un auto-logout avec redirection vers /admin sur code 401, lance une exception Response sur tout code non-ok

Note : les endpoints publics (PasswordChange, AccountUnlock, SecurityQuestionSetup) n'utilisent pas cet utilitaire. Ils font des appels fetch() directs avec credentials: 'include' et le pattern CSRF Sanctum.

Pattern "Document Viewer" — reutilise dans 8+ vues

Un pattern recurrent dans les vues documentaires avec un jeu de fonctionnalites coherent :

  • Barre de recherche : filtrage temps réel sur titre, introduction et tags
  • Filtres par tags : chips cliquables (toggle, clear) pour isoler les documents par theme
  • Tri configurable : par titre (alpha) ou date de mise a jour (recent)
  • Boutons "Ouvrir tout" / "Fermer tout" : bascule globale des accordeons

8.a Authentification — comptes locaux bcrypt

  • Modèle : App\Models\Hr\HrUser — champs login, password (bcrypt), display_name, role (drh | rh), is_active, must_change_password

8.b Reset de mot de passe par lien

Mécanisme inspiré de Laravel Password Broker mais piloté par les DRH eux-mêmes (pas d'envoi d'email automatique — le lien est fourni à l'utilisateur en main propre ou via un canal interne sécurisé) :

  1. Le DRH ouvre la liste des comptes RH dans le dashboard, clique sur "Générer un lien de réinitialisation" sur la fiche d'un compte
  2. Le backend (HrAccountController::generateResetLink) crée une entrée dans hr_password_resets avec un token aléatoire de 64 caractères et une expires_at à +24 h
  3. Le DRH copie le lien https://intranet.cpms.services/rh/reset/<token> et le transmet à l'utilisateur
  4. L'utilisateur ouvre le lien → route /rh/reset/:token → HrResetPassword.vue
  5. Le frontend appelle POST /api/hr/reset/check pour vérifier la validité (throttle 10/min)
  6. L'utilisateur saisit son nouveau mot de passe → POST /api/hr/reset/confirm (throttle 5/min) qui rehache et invalide le token

Les tokens sont consommables une seule fois et expirent au bout de 24 h. Tout token invalide ou expiré renvoie une erreur générique pour ne pas fuiter d'information sur l'existence d'un compte.

9.a Architecture

  • Auth : POST /api/chat/login → ChatAuthService → LdapService::bind avec login Windows + mot de passe Windows → émission d'un token Bearer dans chat_api_tokens
  • Profil : à la première connexion, le frontend appelle GET /api/chat/profile qui crée la fiche dans chat_profiles (display_name dérivé du sAMAccountName, avatar_url null)
  • Avatar : upload via POST /api/chat/profile/avatar (multipart, image), stocké dans storage/app/chat-avatars/
  • Messages : table chat_messages avec sender_login, content, created_at, deleted_at (soft-delete), index sur created_at
  • Polling : GET /api/chat/public/latest renvoie les N derniers messages non supprimés, le frontend rafraîchit toutes les 5 secondes
  • Modération : chaque utilisateur peut supprimer ses propres messages (soft-delete via deleted_at) ; les messages supprimés disparaissent de la liste publique

9.b Charte de modération — affichée à la connexion

Avant de saisir ses identifiants, l'utilisateur doit cocher l'acceptation d'une charte affichée sur ChatView.vue :

  • Respecter les collègues — pas d'insultes, propos discriminatoires ou harcèlement
  • Eviter le spam — pas de messages répétitifs ou hors sujet
  • Protéger la confidentialité — pas de mots de passe ni données clients sensibles
  • Rester professionnel — pas de contenu inapproprié
  • Assumer ses propos — chaque message est associé à l'identité Windows

L'entreprise se réserve le droit de modérer ou supprimer tout message ne respectant pas ces règles.

10. Sécurité

La sécurité couvre l'ensemble des couches de l'application, de l'authentification Active Directory au chiffrement des mots de passe sur le fil.

Authentification et sessions

  • Admin — Bearer Token : authentification AD + vérification du groupe Intranet-Admins, token genere et stocke en BDD (admin_api_tokens), envoye dans l'en-tete Authorization: Bearer
  • Validation triple : le middleware AdminApiAuth vérifié le token, l'adresse IP et le User-Agent a chaque requête
  • CSRF Sanctum : les operations publiques (changement MDP, déblocage) utilisent le pattern cookie CSRF de Laravel Sanctum (/sanctum/csrf-cookie + X-XSRF-TOKEN)
  • Auto-logout : redirection automatique vers /admin sur code HTTP 401 (token expire ou invalide)
  • Credentials isolation : credentials: 'omit' pour les routes admin (token seul), credentials: 'include' pour les routes publiques CSRF

Politique de mot de passe

  • Longueur minimum : 15 caracteres (superieur a la recommandation ANSSI de 12)
  • Complexite obligatoire : au moins 1 majuscule, 1 chiffre, 1 caractere special (@$!%*?&)
  • Jauge de force : indicateur visuel en temps réel (4 barres : Faible / Moyenne / Bonne / Excellente)
  • Checklist interactive : chaque critere se valide visuellement a la saisie
  • Format unicodePwd : encodage UTF-16LE avec guillemets obligatoires pour Active Directory

Questions de sécurité

  • Stockage : réponses jamais en clair, hachees avec un sel unique (salt) par réponse
  • Anti-copie : copier, couper et coller desactives sur les champs de réponse (@copy.prevent @paste.prevent @cut.prevent)
  • Effacement automatique : la réponse et le mot de passe sont effaces lorsque l'utilisateur quitte la page (visibilitychange, pagehide)
  • Journalisation : chaque tentative (reussie ou echouee) est enregistree avec IP dans security_questions_attempts
  • Piste d'audit : création, modification et suppression de questions tracees dans security_questions_audit

Sécurité LDAP

  • LDAPS : connexion chiffree TLS sur le port 636 (jamais en clair sur 389)
  • Comptes de service dedies : un compte pour le changement de mot de passe (permission "Reset Password") et un compte pour le déblocage uniquement (permission "Unlock Account"), principe du moindre privilege
  • Format NetBIOS : comptes de service au format DOM-AD\\USERNAME
  • Certificat CA : import du certificat LDAPS dans le trust store système (/usr/local/share/ca-certificates/)

Sécurité AS400/ODBC

  • Variable d'environnement : l'ancien mot de passe est passe via AS400_OLD_PASSWORD (jamais en argument de commande)
  • Echappement : identifiant et nouveau mot de passe echappes via escapeshellarg()
  • Isolation reseau : le middleware ODBC n'est pas expose publiquement
  • Distinction des erreurs : differenciation entre echec de connexion et echec d'authentification dans les logs

En-tetes et bonnes pratiques frontend

  • cache: 'no-store' : aucune mise en cache des réponses API sensibles
  • mode: 'cors' : requêtes cross-origin explicites
  • <meta robots noindex,nofollow> : exclusion des moteurs de recherche
  • HttpOnly cookies : les cookies CSRF Sanctum sont inaccèssibles au JavaScript client
  • AbortController : annulation propre des requêtes en cours lors du demontage du composant
