1. Introduction à Docker

Qu'est-ce que Docker ?

Docker est une plateforme open source qui permet de créer, déployer et exécuter des applications dans des conteneurs. Un conteneur est une unité logicielle légère et portable qui embarque tout le nécessaire pour faire fonctionner une application : code source, runtime, bibliothèques système et paramètres de configuration.

Lancé en 2013, Docker a révolutionné le développement logiciel en permettant de garantir que « si ça marche sur ma machine, ça marchera partout ». Cette reproductibilité élimine les problèmes classiques d'incompatibilité entre les environnements de développement, de test et de production.

Docker vs Machines Virtuelles

CritèreMachine VirtuelleConteneur Docker
IsolationOS complet virtualiséProcessus isolé sur le noyau hôte
TaillePlusieurs Go (OS + app)Quelques Mo à centaines de Mo
DémarrageMinutesSecondes
PerformanceOverhead de l'hyperviseurQuasi-native
PortabilitéLiée à l'hyperviseurPortable partout où Docker tourne
DensitéQuelques VM par serveurDes centaines de conteneurs par serveur

Astuce : Les conteneurs et les VM ne s'excluent pas mutuellement. En production, il est courant d'exécuter des conteneurs Docker à l'intérieur de machines virtuelles pour combiner isolation matérielle et légèreté applicative.

Architecture de Docker Engine

Docker repose sur une architecture client-serveur composée de trois composants principaux :

  • Docker Daemon (dockerd) — Le processus serveur qui gère les objets Docker (images, conteneurs, réseaux, volumes). Il écoute les requêtes de l'API Docker.
  • Docker CLI (docker) — L'interface en ligne de commande utilisée par le développeur pour interagir avec le daemon via l'API REST.
  • Docker Registry — Un service de stockage et de distribution d'images Docker. Docker Hub est le registry public par défaut.
Architecture
+------------------+       API REST       +------------------+
|   Docker CLI     | -------------------> |  Docker Daemon   |
|   (docker)       |                      |  (dockerd)       |
+------------------+                      +--------+---------+
                                                   |
                                          +--------+---------+
                                          |   Conteneurs     |
                                          |   Images         |
                                          |   Volumes        |
                                          |   Réseaux       |
                                          +------------------+
                                                   |
                                          +--------+---------+
                                          | Docker Registry  |
                                          | (Docker Hub)     |
                                          +------------------+

Image vs Conteneur

La distinction fondamentale à comprendre :

  • Image — Un modèle en lecture seule contenant les instructions pour créer un conteneur. C'est comme un moule ou un plan de construction.
  • Conteneur — Une instance exécutable d'une image. C'est l'objet vivant qui tourne sur la machine. On peut créer plusieurs conteneurs à partir d'une même image.

2. Installation

Windows (Docker Desktop + WSL2)

Docker Desktop pour Windows utilise WSL2 (Windows Subsystem for Linux 2) comme backend, offrant de bien meilleures performances que l'ancien backend Hyper-V.

  1. Activer WSL2 dans les fonctionnalités Windows (PowerShell en admin) :
PowerShell (Admin)
wsl --install
wsl --set-default-version 2
  1. Télécharger et installer Docker Desktop depuis docker.com
  2. Dans les paramètres de Docker Desktop, activer « Use the WSL 2 based engine »
  3. Redémarrer la machine si nécessaire

Linux (Ubuntu / Debian)

Bash
# Mise à jour des paquets
sudo apt update && sudo apt upgrade -y

# Installation des prérequis
sudo apt install -y ca-certificates curl gnupg lsb-release

# Ajout de la clé GPG officielle Docker
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# Ajout du dépôt Docker
echo "deb [arch=$(dpkg --print-architecture) \
  signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Installation de Docker Engine
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io \
  docker-buildx-plugin docker-compose-plugin

# Ajouter l'utilisateur au groupe docker (pour éviter sudo)
sudo usermod -aG docker $USER

Attention : Après avoir ajouté votre utilisateur au groupe docker, vous devez vous déconnecter et reconnecter (ou lancer newgrp docker) pour que le changement prenne effet.

Vérification de l'installation

Terminal
# Vérifier la version installée
docker version

# Afficher les informations système Docker
docker info

# Tester avec le conteneur hello-world
docker run hello-world

Astuce : La commande docker run hello-world télécharge automatiquement l'image si elle n'est pas présente localement, la lance, affiche un message de confirmation, puis le conteneur s'arrête.

3. Concepts fondamentaux

Docker s'articule autour de cinq objets fondamentaux qu'il est essentiel de maîtriser :

Images

Modèles en lecture seule constitués de couches superposées (layers). Chaque instruction d'un Dockerfile crée une nouvelle couche. Les couches sont mises en cache pour accélérer les builds.

Conteneurs

Instances exécutables d'images. Ils ajoutent une couche inscriptible au-dessus de l'image. Leur état est éphémère par défaut (supprimé avec le conteneur).

Volumes

Mécanisme de persistance des données géré par Docker. Les volumes survivent à la suppression des conteneurs et peuvent être partagés entre plusieurs conteneurs.

Réseaux

Permettent la communication entre conteneurs de manière isolée. Docker crée par défaut un réseau bridge pour chaque conteneur.

Registries

Services de stockage et de distribution d'images Docker. Docker Hub est le registry public par défaut, mais on peut utiliser des registries privés (GitHub Container Registry, AWS ECR, GitLab Registry, etc.).

Cycle de vie d'un conteneur

Lifecycle
  docker create        docker start        docker stop
[Image] --------> [Created] --------> [Running] --------> [Stopped]
                      |                    |   ^               |
                      |    docker restart   |   |               |
                      |                    +---+               |
                      |                                        |
                      |           docker rm                    |
                      +----> [Deleted] <-----------------------+

États possibles :
  - Created  : conteneur créé mais pas encore lancé
  - Running  : conteneur en cours d'exécution
  - Paused   : conteneur mis en pause (docker pause)
  - Stopped  : conteneur arrêté proprement
  - Deleted  : conteneur supprimé définitivement

5. Dockerfile

Un Dockerfile est un fichier texte contenant les instructions pour construire une image Docker. Chaque instruction crée une couche (layer) dans l'image finale.

Instructions principales

InstructionRôle
FROMImage de base (première instruction obligatoire)
WORKDIRDéfinir le répertoire de travail dans le conteneur
COPYCopier des fichiers de l'hôte vers le conteneur
RUNExécuter une commande lors du build
EXPOSEDocumenter le port utilisé (informatif)
ENVDéfinir une variable d'environnement
CMDCommande par défaut au lancement du conteneur
ENTRYPOINTPoint d'entrée du conteneur (non remplaçable facilement)
ARGVariable de build uniquement (pas disponible au runtime)
VOLUMEDéclarer un point de montage de volume

Exemple : Application Node.js

Dockerfile
# Image de base légère avec Node.js 20
FROM node:20-alpine

# Créer le répertoire de travail
WORKDIR /app

# Copier les fichiers de dépendances d'abord (cache Docker)
COPY package.json package-lock.json ./

# Installer les dépendances
RUN npm ci --only=production

# Copier le reste du code source
COPY . .

# Documenter le port utilisé
EXPOSE 3000

# Définir une variable d'environnement
ENV NODE_ENV=production

# Créer un utilisateur non-root pour la sécurité
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Commande de lancement
CMD ["node", "server.js"]

Exemple : Application Python / Flask

Dockerfile
# Image de base Python slim
FROM python:3.12-slim

# Définir le répertoire de travail
WORKDIR /app

# Installer les dépendances système nécessaires
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    && rm -rf /var/lib/apt/lists/*

# Copier et installer les dépendances Python
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copier le code source
COPY . .

# Créer un utilisateur non-root
RUN useradd -m -r appuser
USER appuser

# Exposer le port Flask
EXPOSE 5000

# Variable d'environnement Flask
ENV FLASK_APP=app.py
ENV FLASK_ENV=production

# Lancer l'application avec gunicorn
CMD ["gunicorn", "--bind", "0.0.0.0:5000", "--workers", "4", "app:app"]

Multi-stage builds

Les builds multi-étapes permettent de réduire la taille de l'image finale en séparant l'étape de compilation de l'étape d'exécution.

Dockerfile (multi-stage)
# Étape 1 : Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Étape 2 : Production (image finale légère)
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Astuce : L'image finale ne contient que Nginx et les fichiers buildés. Toutes les dépendances de build (Node.js, npm, node_modules) sont exclues, réduisant la taille de l'image de plusieurs centaines de Mo à quelques dizaines de Mo.

Fichier .dockerignore

Le fichier .dockerignore fonctionne comme .gitignore et empêche la copie de fichiers inutiles dans le contexte de build :

.dockerignore
node_modules
npm-debug.log
.git
.gitignore
.env
.env.*
Dockerfile
docker-compose.yml
README.md
.vscode
coverage
dist
.DS_Store

Bonnes pratiques Dockerfile

  • Exploiter le cache de couches — Copier package.json avant le code source pour que npm install ne soit relancé que si les dépendances changent.
  • Utiliser des images de base minimales — Préférer alpine ou slim aux images complètes (ex: node:20-alpine fait ~50 Mo vs ~350 Mo pour node:20).
  • Exécuter en non-root — Toujours créer et utiliser un utilisateur dédié via USER.
  • Combiner les commandes RUN — Réduire le nombre de couches en chaînant les commandes avec &&.
  • Nettoyer dans la même couche — Supprimer les caches et fichiers temporaires dans le même RUN que l'installation.

6. Docker Compose

Docker Compose est un outil qui permet de définir et gérer des applications multi-conteneurs à l'aide d'un fichier YAML. Au lieu de lancer chaque conteneur individuellement avec de longues commandes docker run, on décrit toute l'infrastructure dans un seul fichier docker-compose.yml.

Exemple complet : Stack Web + MySQL + phpMyAdmin

docker-compose.yml
version: "3.8"

services:
  # ---- Application Web (Node.js) ----
  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: mon-app-web
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - DB_HOST=db
      - DB_PORT=3306
      - DB_USER=app_user
      - DB_PASSWORD=secret123
      - DB_NAME=ma_base
    volumes:
      - ./src:/app/src       # Bind mount pour le dev (hot reload)
    depends_on:
      db:
        condition: service_healthy
    networks:
      - app-network
    restart: unless-stopped

  # ---- Base de données MySQL ----
  db:
    image: mysql:8.0
    container_name: mon-app-db
    ports:
      - "3306:3306"
    environment:
      MYSQL_ROOT_PASSWORD: rootpassword
      MYSQL_DATABASE: ma_base
      MYSQL_USER: app_user
      MYSQL_PASSWORD: secret123
    volumes:
      - db-data:/var/lib/mysql          # Volume nommé pour persistance
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # Script d'init
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - app-network
    restart: unless-stopped

  # ---- phpMyAdmin ----
  phpmyadmin:
    image: phpmyadmin:latest
    container_name: mon-app-pma
    ports:
      - "8080:80"
    environment:
      PMA_HOST: db
      PMA_PORT: 3306
      UPLOAD_LIMIT: 64M
    depends_on:
      - db
    networks:
      - app-network
    restart: unless-stopped

# ---- Volumes ----
volumes:
  db-data:
    driver: local

# ---- Réseaux ----
networks:
  app-network:
    driver: bridge

CommandeDescription

docker compose down -vIdem + supprimer les volumes

Variables d'environnement avec .env

Pour éviter de stocker des secrets directement dans le docker-compose.yml, utilisez un fichier .env :

.env
MYSQL_ROOT_PASSWORD=rootpassword
MYSQL_DATABASE=ma_base
MYSQL_USER=app_user
MYSQL_PASSWORD=secret123
NODE_ENV=production

Puis référencez les variables dans le fichier Compose :

docker-compose.yml (extrait)
services:
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_DATABASE: ${MYSQL_DATABASE}
      MYSQL_USER: ${MYSQL_USER}
      MYSQL_PASSWORD: ${MYSQL_PASSWORD}

Attention : Ne versionnez jamais le fichier .env contenant des secrets dans Git. Ajoutez-le à votre .gitignore et fournissez un fichier .env.example avec des valeurs factices.

7. Volumes et persistance

Par défaut, les données écrites dans un conteneur sont éphémères : elles disparaissent à la suppression du conteneur. Docker propose deux mécanismes pour persister les données.

Named Volumes vs Bind Mounts

CritèreNamed VolumeBind Mount
Géré parDockerL'utilisateur (système de fichiers hôte)
Emplacement/var/lib/docker/volumes/N'importe où sur l'hôte
PerformanceOptimale sur tous les OSVariable (lent sur macOS/Windows)
Cas d'usageDonnées de BDD, fichiers persistantsCode source en dev, fichiers de config
BackupVia commandes DockerAccès direct au système de fichiers
Syntaxe-v mon-volume:/data-v ./local:/data

Commandes de gestion des volumes

Terminal
# Créer un volume nommé
docker volume create mes-donnees

# Lister tous les volumes
docker volume ls

# Inspecter un volume (voir son chemin réel)
docker volume inspect mes-donnees

# Supprimer un volume
docker volume rm mes-donnees

# Supprimer tous les volumes non utilisés
docker volume prune

Utilisation dans docker run

Terminal
# Named volume : les données MySQL persistent
docker run -d \
  --name ma-bdd \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  mysql:8.0

# Bind mount : synchroniser le code local avec le conteneur
docker run -d \
  --name mon-dev \
  -v $(pwd)/src:/app/src \
  -p 3000:3000 \
  node:20-alpine npm run dev

Stratégie de backup d'un volume

Terminal
# Sauvegarder un volume dans une archive tar
docker run --rm \
  -v mysql-data:/source:ro \
  -v $(pwd)/backups:/backup \
  alpine tar czf /backup/mysql-backup-$(date +%Y%m%d).tar.gz -C /source .

# Restaurer un volume depuis une archive
docker run --rm \
  -v mysql-data:/target \
  -v $(pwd)/backups:/backup \
  alpine tar xzf /backup/mysql-backup-20250315.tar.gz -C /target

Astuce : Pour les bases de données, privilégiez les outils natifs (mysqldump, pg_dump) pour les backups plutôt que de copier les fichiers bruts du volume, afin de garantir la cohérence des données.

8. Réseau Docker

Docker gère automatiquement le réseau de ses conteneurs. Comprendre les différents types de réseaux est essentiel pour architecturer correctement ses applications.

Types de réseaux

DriverDescriptionCas d'usage
bridgeRéseau isolé par défaut. Les conteneurs sur le même bridge communiquent entre eux.Environnement local, multi-conteneurs
hostLe conteneur partage le réseau de l'hôte directement. Pas d'isolation réseau.Performance maximale, debugging
noneAucun accès réseau. Le conteneur est totalement isolé.Traitements offline, sécurité
overlayRéseau distribué entre plusieurs hôtes Docker (Swarm).Cluster Docker Swarm

Communication inter-conteneurs

Dans un réseau bridge personnalisé, les conteneurs peuvent se joindre par leur nom grâce au DNS interne de Docker :

Terminal
# Créer un réseau personnalisé
docker network create mon-reseau

# Lancer un conteneur MySQL sur ce réseau
docker run -d \
  --name ma-bdd \
  --network mon-reseau \
  -e MYSQL_ROOT_PASSWORD=secret \
  mysql:8.0

# Lancer un conteneur Node.js sur le même réseau
# Il peut accéder à MySQL via le nom "ma-bdd"
docker run -d \
  --name mon-app \
  --network mon-reseau \
  -e DB_HOST=ma-bdd \
  -p 3000:3000 \
  mon-app-image

Astuce : Utilisez toujours un réseau bridge personnalisé plutôt que le bridge par défaut. Le bridge par défaut ne supporte pas la résolution DNS par nom de conteneur — seules les adresses IP fonctionnent.

Mapping de ports

Le mapping de ports permet d'exposer un service du conteneur vers l'hôte :

Terminal
# Syntaxe : -p [IP_HOTE:]PORT_HOTE:PORT_CONTENEUR[/PROTOCOLE]

# Mapper le port 80 du conteneur vers le port 8080 de l'hôte
docker run -p 8080:80 nginx

# Mapper uniquement sur localhost (non accessible depuis l'extérieur)
docker run -p 127.0.0.1:8080:80 nginx

# Mapper un port UDP
docker run -p 53:53/udp dns-server

# Laisser Docker choisir un port hôte aléatoire
docker run -p 80 nginx
docker port mon-conteneur   # voir le port attribué

Commandes réseau

Terminal
# Lister les réseaux
docker network ls

# Inspecter un réseau (voir les conteneurs connectés)
docker network inspect mon-reseau

# Connecter un conteneur existant à un réseau
docker network connect mon-reseau mon-conteneur

# Déconnecter un conteneur d'un réseau
docker network disconnect mon-reseau mon-conteneur

# Supprimer un réseau
docker network rm mon-reseau

# Supprimer les réseaux non utilisés
docker network prune

9. Bonnes pratiques

Sécurité

  • Ne jamais exécuter en root — Utiliser USER dans le Dockerfile pour créer un utilisateur dédié. Un conteneur compromis avec les droits root peut accéder à l'hôte.
  • Scanner les images — Utiliser docker scout ou des outils comme Trivy pour détecter les vulnérabilités dans les images.
  • Utiliser des images officielles — Privilégier les images marquées « Docker Official Image » sur Docker Hub.
  • Pinner les versions — Utiliser node:20.11-alpine plutôt que node:latest pour des builds reproductibles.
  • Ne pas stocker de secrets dans l'image — Utiliser des variables d'environnement, des fichiers .env ou Docker Secrets.
  • Limiter les capabilities — Utiliser --cap-drop ALL et n'ajouter que les capabilities nécessaires.
Terminal
# Scanner une image pour les vulnérabilités
docker scout cves mon-app:latest

# Lancer avec des capabilities réduites
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE nginx

Optimisation des images

  • Choisir des images de base légères — Comparer les tailles :
ImageTaille approximative
ubuntu:22.04~77 Mo
debian:bookworm-slim~74 Mo
alpine:3.19~7 Mo
node:20~350 Mo
node:20-alpine~50 Mo
python:3.12~390 Mo
python:3.12-slim~45 Mo
  • Exploiter le cache de couches — Ordonner les instructions du plus stable au plus variable.
  • Combiner les commandes RUN — Moins de couches = image plus légère.
  • Utiliser les multi-stage builds — Séparer la compilation de l'exécution.
  • Toujours utiliser .dockerignore — Exclure node_modules, .git, fichiers temporaires.

Limites de ressources

Terminal
# Limiter la mémoire à 512 Mo et les CPUs à 1.5
docker run -d \
  --memory=512m \
  --cpus=1.5 \
  --name mon-app \
  mon-image

# Avec Docker Compose
# services:
#   app:
#     deploy:
#       resources:
#         limits:
#           cpus: "1.5"
#           memory: 512M
#         reservations:
#           cpus: "0.5"
#           memory: 256M

Logging

Terminal
# Voir les logs en temps réel
docker logs -f mon-conteneur

# Limiter aux 100 dernières lignes
docker logs --tail 100 mon-conteneur

# Logs avec horodatage
docker logs -t mon-conteneur

# Configurer la rotation des logs (docker run)
docker run -d \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  mon-app

Mettons en pratique l'ensemble des concepts vus dans ce guide en construisant une stack LAMP (Linux, Apache, PHP, MySQL) complète avec Docker Compose, idéale pour le développement web en BTS SIO.

Étape 5 : docker-compose.yml complet

docker-compose.yml
version: "3.8"

services:
  # ---- Apache + PHP ----
  web:
    build:
      context: .
      dockerfile: php/Dockerfile
    container_name: lamp-web
    ports:
      - "${WEB_PORT:-8080}:80"
    volumes:
      - ./src:/var/www/html          # Code source monté
      - ./config/php.ini:/usr/local/etc/php/conf.d/custom.ini
    depends_on:
      db:
        condition: service_healthy
    environment:
      - DB_HOST=db
      - DB_NAME=${MYSQL_DATABASE}
      - DB_USER=${MYSQL_USER}
      - DB_PASS=${MYSQL_PASSWORD}
    networks:
      - lamp-network
    restart: unless-stopped

  # ---- MySQL 8.0 ----
  db:
    image: mysql:8.0
    container_name: lamp-db
    ports:
      - "${DB_PORT:-3306}:3306"
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_DATABASE: ${MYSQL_DATABASE}
      MYSQL_USER: ${MYSQL_USER}
      MYSQL_PASSWORD: ${MYSQL_PASSWORD}
    volumes:
      - db-data:/var/lib/mysql
      - ./sql:/docker-entrypoint-initdb.d
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    networks:
      - lamp-network
    restart: unless-stopped

  # ---- phpMyAdmin ----
  phpmyadmin:
    image: phpmyadmin:latest
    container_name: lamp-pma
    ports:
      - "${PMA_PORT:-8081}:80"
    environment:
      PMA_HOST: db
      PMA_PORT: 3306
      PMA_USER: root
      PMA_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      UPLOAD_LIMIT: 64M
    depends_on:
      - db
    networks:
      - lamp-network
    restart: unless-stopped

volumes:
  db-data:
    driver: local

networks:
  lamp-network:
    driver: bridge

Étape 7 : Lancer et tester

Terminal
# Se placer dans le répertoire du projet
cd mon-projet-lamp

# Construire et lancer la stack
docker compose up -d --build

# Vérifier que les 3 services tournent
docker compose ps

# Voir les logs en temps réel
docker compose logs -f

# Accéder aux services :
#   Application web  : http://localhost:8080
#   phpMyAdmin       : http://localhost:8081

# Se connecter au conteneur PHP pour debug
docker compose exec web bash

# Se connecter directement à MySQL
docker compose exec db mysql -u lamp_user -p lamp_app

# Arrêter la stack (conserve les données)
docker compose down

# Arrêter et supprimer les données
docker compose down -v

Astuce : Utilisez docker compose up -d --build pour reconstruire les images modifiées. Sans le flag --build, Docker Compose réutilise les images en cache même si le Dockerfile a changé.
